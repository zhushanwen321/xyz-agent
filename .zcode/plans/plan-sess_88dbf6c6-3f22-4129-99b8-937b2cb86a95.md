## 问题根因

Electron GUI 应用从 macOS LaunchServices 启动时，`process.env.PATH` 是最小值（`/usr/bin:/bin:/usr/sbin:/sbin`），缺少 `~/.local/bin`、`~/.cargo/bin`、`/opt/homebrew/bin` 等用户级 bin 目录。这个残缺 PATH 经 `buildSafeEnv`（白名单保留 PATH 但不补全）→ runtime 子进程 → `buildSafeEnv` 二次过滤 → pi 子进程 → pi `getShellEnv()`（只 prepend `~/.pi/agent/bin`，不 source 任何 profile），一路原样传递。pi 的 bash 工具最终拿到残缺 PATH，找不到 `uv`。

pi 原生 CLI 模式无此问题——从终端启动时继承了登录 shell 的完整 PATH。

## 修复方案（长期）

在 Electron 主进程最早期调用 `fixPathEnv()`，用用户登录 shell（`$SHELL -ilc 'env'`）读取完整 PATH，赋给 `process.env.PATH`。后续 `buildSafeEnv` 自然把完整 PATH 传递到 pi。一处修复，全链路受益。

### 自实现，不引 npm 包

`shell-env`/`fix-path` 等包的核心逻辑就是 `spawnSync($SHELL, ['-ilc', 'env'])`，自实现 ~40 行。不引包的理由：
- 引包要同时改 `package.json` deps + `vite.config.main.ts` external + `electron-builder.yml` files + preflight-check.sh，碰打包链路代价大（规则 #12 打包约束高度敏感）
- `spawnSync` + 解析 `KEY=VALUE` 逻辑足够简单，零依赖可控

### 改动清单

**1. 新建 `apps/electron/main/supervisor/shell-env.ts`**

导出 `fixPathEnv()`：
- Windows 跳过（GUI 应用从注册表读取 PATH，通常完整）
- `$SHELL -ilc 'env'` 读取登录 shell 完整环境（`-l` login 加载 `.zprofile`/`.bash_profile`，`-i` interactive 加载 `.zshrc`/`.bashrc`，两者都加载才能覆盖用户在任意配置文件中设置的 PATH）
- `spawnSync` + `timeout: 5000`（用户 shell 配置异常时防卡死）
- 解析 stdout 只认 `^[A-Za-z_][A-Za-z0-9_]*=` 格式的行（跳过 motd/fortune 等污染输出）
- 提取 shell 的 PATH，**只在比当前 PATH 更长时覆盖**（安全：终端启动时 PATH 已完整，覆盖无害；GUI 启动时 shell PATH 更长，覆盖生效；shell 配置异常返回短 PATH 时不破坏已有的）
- `spawnSync` 失败或超时不修改 `process.env`（fail-safe）

**2. 修改 `apps/electron/main/main.ts`**

在 EPIPE 兜底（L76）之后、`isDev` 判断（L78）之前插入：
```ts
import { fixPathEnv } from './supervisor/shell-env.js'
// ...
// ── PATH 修复（GUI 启动时补全用户级 bin 目录）──────────────────────
fixPathEnv()
```

这是最早可执行位置，确保后续所有代码（`buildSafeEnv`、`process-control.ts` spawn）都使用完整 PATH。

**3. 新建 `apps/electron/main/test/shell-env.test.ts`**

测试 `fixPathEnv()` 行为（mock `spawnSync`）：
- shell 返回更长 PATH → `process.env.PATH` 被更新
- shell 返回更短 PATH → 不覆盖
- `spawnSync` 失败（非零退出）→ 不修改
- `spawnSync` 超时 → 不修改
- Windows 平台 → 跳过（不调 spawnSync）
- stdout 含 motd 污染行 → 只解析 `KEY=VALUE` 行

### 不改动的部分

- `safe-env.ts` / `constants.ts` 白名单——PATH 已在白名单里，修复后自然传递
- `process-control.ts` / `rpc-client.ts`——`buildSafeEnv` 逻辑不变
- `process-manager.ts:180` piDir prepend——在完整 PATH 基础上 prepend，结果正确
- pi 源码——pi `getShellEnv()` `{...process.env}` 继承修复后的完整 PATH

### 链路验证（修复后）

```
main.ts fixPathEnv()           → process.env.PATH = 完整 PATH（含 ~/.local/bin 等）
  ↓ buildSafeEnv (白名单保留 PATH)
runtime 子进程                  → process.env.PATH = 完整 PATH
  ↓ buildSafeEnv (二次过滤, PATH 仍保留)
pi 子进程                       → process.env.PATH = 完整 PATH
  ↓ pi getShellEnv() (只 prepend ~/.pi/agent/bin)
pi bash 工具                    → PATH = ~/.pi/agent/bin:完整PATH ✓
```
