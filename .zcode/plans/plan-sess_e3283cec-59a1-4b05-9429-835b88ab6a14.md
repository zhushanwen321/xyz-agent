# CwStore JSON 化改造计划

## 目标

把 coding-workflow 的 `CwStore` 持久化从 `node:sqlite` 换成 JSON 文件，使 pi 的 Bun 编译 binary 能加载该 extension。

## 改动边界

| 文件 | 改动性质 | 说明 |
|------|---------|------|
| `src/cw/store.ts` | **全量重写** | sqlite → JSON 持久化层 + 内存事务 + 文件锁。公开方法签名不变 |
| `src/index.ts` | 1 行 | `resolveCwDbPath` 返回 `.json` 后缀（`_cw.json`） |
| `src/cw/actions/__tests__/_helpers.ts` | 删 1 行 | 删除 `import { DatabaseSync } from "node:sqlite"`（已无用） |
| `src/cw/__tests__/store.test.ts` | 适配 | 去掉 sqlite 专属断言（PRAGMA/SQL 直查），保留行为断言，加事务原子性测试 |
| `src/cw/types.ts` / `state-machine.ts` / `gates.ts` / `actions/*.ts` | **零改动** | 接口不变式 |

## 设计要点

### 1. 存储格式

路径：`~/.pi/agent/cw/<encoded-cwd>/_cw.json`（与原 `_cw.db` 同目录）

```typescript
interface CwJsonFile {
  schemaVersion: number   // 4，与原 SCHEMA_VERSION 对齐
  topics: TopicRecord[]   // 领域模型直接序列化（无 snake_case 转换）
  waves: WaveRecord[]
  testCases: TestCaseRecord[]
  gateHistory: GateHistoryRecord[]
}
```

每个 Record 就是 `CwTopic`/`Wave`/`TestCase`/`GateHistoryEntry` 字段的直接 JSON 序列化，额外加 `topicId` 外键（waves/testCases/gateHistory 记录所属 topic）。比 sqlite 行映射更简单——无 prepare/SQL/参数绑定/列名转换。

空库初始值：`{ schemaVersion: 4, topics: [], waves: [], testCases: [], gateHistory: [] }`

### 2. 事务实现（原子性等价恢复）

核心：transaction 回调内的所有写操作都作用在**内存深拷贝**上。回调正常结束 → 原子落盘；回调抛异常 → 丢弃副本（等价 ROLLBACK）。

```typescript
transaction<T>(fn: () => T): T {
  this.acquireLock()                         // 排他文件锁
  const snapshot = this.loadFileData()       // 读磁盘
  this.fileData = structuredClone(snapshot)  // 深拷贝（事务隔离）
  this.inTransaction = true
  try {
    const result = fn()                      // 业务操作内存对象（read-after-write 天然一致）
    this.flushToDisk()                       // write temp + fsync + rename
    return result
  } catch (err) {
    this.fileData = snapshot                 // ROLLBACK：丢弃副本
    throw err
  } finally {
    this.inTransaction = false
    this.releaseLock()
  }
}
```

事务外的读方法（如 `loadTopic` 在 `transaction` 之前调用）：`getActiveData()` 判断——事务内返回 `this.fileData`，事务外读磁盘。

写方法必须在事务内调用（`ensureWritable()`），否则 throw。现有 9 个 action handler 全部用 `transaction(() => {...})` 包裹，满足此约束。

### 3. 文件锁（跨进程并发安全）

并发来源：pi workflow extension 的 `parallel()` 派发多个 sub-agent，每个独立调 CW tool，可能跨进程并发访问同一 `_cw.json`。

锁机制：lockfile + O_EXCL 原子创建（不依赖 `flock`，Bun 支持不确定）。

```
_cw.json       ← 数据
_cw.json.lock  ← 锁文件（内容 = PID + 时间戳）
```

- `openSync(lockPath, 'wx')`：O_EXCL 原子创建，已存在抛 EEXIST
- 退避重试：100ms × 50 次 = 总等待上限 5s（与原 sqlite `busy_timeout=5000` 对齐）
- stale lock 检测：`process.kill(pid, 0)` 探活 + 30s 超时兜底
- break stale lock：`unlinkSync(lockPath)` 后重试 O_EXCL

### 4. 原子写入（崩溃一致性）

```
writeFileSync(tmpPath, json)   ← 写临时文件 _cw.json.tmp
fsyncSync(tmpFd)               ← 数据落盘（不止 page cache）
renameSync(tmpPath, dbPath)    ← POSIX 原子 rename
fsyncSync(dirFd)               ← dir entry 持久化
```

任一阶段 crash，磁盘上要么旧文件完整要么新文件完整，不会半个。

### 5. schema 演进

JSON 方案字段直接存在领域对象上。旧 JSON（schemaVersion < 4）加载时补默认值（`topicDir=""`、`requiresScreenshot=false`、`dependsOn=[]`）。不做 sqlite → JSON 自动迁移（用户确认丢弃旧数据）。
