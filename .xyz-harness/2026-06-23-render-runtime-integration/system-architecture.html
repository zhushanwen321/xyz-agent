系统架构 — 前端 renderer ↔ runtime 集成（W11+）

来源：system-architecture.md · 主角图：分层架构图 + 状态机图 · verdict: pass / mode: refactor

核心计算：消息路由与状态同步。 系统是消息管道——前端发命令、runtime 处理、状态变更推回前端。核心复杂度在「三类消息形态的统一处理」与「session 隔离的事件路由」。
分层决策：三层架构（Interface / Engine / Infrastructure），非 DDD 四层——无复杂领域规则（如金融计算/库存扣减），四层会多出空壳 domain 层。
终态不可逆 派生展示态 瞬态/事件流

4. 状态机图

4.1 Session 生成状态 终态 idle 不可逆

stateDiagram-v2 [*] --> idle idle --> generating: message.send generating --> idle: abort generating --> idle: complete

Status 枚举：idle / generating · 终态：idle（不可逆：一旦 idle，只能通过 message.send 再次进入 generating） · Reason 字段：无（终态 idle 无需原因）

4.2 Extension 安装状态 终态 {completed, error} 不可逆

stateDiagram-v2 [*] --> idle idle --> installing: install / installDir / installGit installing --> discovered: (多步流) discovered --> completed: finishInstall discovered --> error: cancelInstall / 失败 completed --> [*] error --> [*]

Status 枚举：idle / installing / discovered / completed / error · 终态集合：{completed, error}（不可逆） · Reason 字段：error 时附带 message · 重试：error 终态「不可逆」指「该次安装流程不可重置」，用户可发起新的安装（新流程）来重试。

4.3 Compact 压缩状态 瞬态，非持久状态机（事件流）

stateDiagram-v2 [*] --> idle idle --> compacting: compact() compacting --> compacted: (原子转换) compacted --> idle

Status 枚举：compacting / compacted · 非真正状态机，而是事件流。compacting→compacted 是原子转换，失败时 compacted 附带 error 字段。

4.4 git-zone 四态 派生展示态，非状态机

flowchart LR R["GitStatusResult
(每次 git.status 返回)"] --> DER["deriveState (重算)"] DER --> C["clean
staged==0 && unstaged==0 && !conflict"] DER --> S["staged
staged>0 && unstaged==0 && !conflict"] DER --> D["dirty
unstaged>0 && !conflict"] DER --> CF["conflict

git-zone 四态从 GitStatusResult 实时推导，无显式转换规则。git-zone 是「镜子」（反映真实状态），不是「控制器」（驱动状态转换）。
