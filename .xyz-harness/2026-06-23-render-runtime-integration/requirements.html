1. 业务目标（Business Goals）

目标树

  • G1 已实装渲染能力在开发联调时可见、可验证 — 成功标准：mock 发消息看到完整实时流式效果（思考→工具→文本→变更→系统通知）
    • G1.1 steer/followup 排队、自动重试这两类「已入库没 UI」的运行时状态，界面上有指示位
  • G2 后端已就绪但前端闲置的能力全部接通 — 成功标准：Extension 安装/卸载、上下文压缩、终端/浏览器 widget 三项有完整入口与闭环
    • G2.1 右抽屉 Side Drawer 作为承载 widget 的架构容器落地
  • G3 还原 v3 设计稿的完整工作面板 — 成功标准：Panel 恢复 5 个固定 zone（含被误删的 git-zone），会话列表支持实时刷新，侧边文件视图显示真实改动
  • G4 契约对齐与收尾 — 成功标准：剩余类型/枚举契约裂缝补齐

2. 业务用例（Use Cases）

用例图见上方 TL;DR hero 区。以下为每个用例的主流程 / 边界 / 异常。

UC-1: 用户看到完整的 AI 工作回合

  • Actor: 日常用户（含开发者） · 关联: G1
  • 主流程: 发送消息 → 思考（折叠） → 调用工具（工具卡） → 流式文本 → 本回合文件变更清单 → 系统通知 → 完成
  • 替代: 工作中 steer 排队引导（不打断）；完成后 followup 开新一轮
  • 异常: 流式出错（红框）；触发自动重试（显示重试指示）
  • 后置: 回合默认折叠，仅显示总结 + 文件变更

UC-2: 用户管理 Extension（安装/卸载）

  • Actor: 日常用户 · 关联: G2
  • 主流程: Settings → Extension → 选来源（npm/目录/git） → 安装 →（目录/git）展示候选 → 确认选中 → 完成，列表刷新
  • 替代: 目录/git 需多步发现候选；可中途取消
  • 异常: 安装失败（回显错误与提示）；非候选目录
  • 后置: 新扩展出现在列表，可启用/禁用

UC-3: 用户主动压缩上下文

  • Actor: 日常用户 · 关联: G2
  • 主流程: slash command 触发 → 显示「压缩中」 → 完成后状态消失 + 出现压缩摘要系统行
  • 异常: 压缩失败（错误回显）
  • 后置: 上下文窗口释放；摘要行留在消息流

UC-4: 开发者在 mock 模式验证流式渲染

  • Actor: 开发者 · 关联: G1
  • 主流程: VITE_MOCK 发任意消息 → mock 按固定剧本回放完整事件序列 → 核对每类渲染符合设计
  • 替代: 中途 abort，剧本应干净停止不残留
  • 后置: 验证通过，进入真实 runtime 联调

UC-5: 用户使用终端/浏览器 widget

  • Actor: 日常用户 · 关联: G2（含 G2.1）
  • 主流程: 运行时推送 widget 内容（按会话通道） → 前端订阅渲染 → 右抽屉对应 tab 呈现
  • 异常: widget 无内容（空态）
  • 后置: widget 持续在抽屉呈现，直到会话切换/关闭

UC-6: 用户查看并操作 git 状态

  • Actor: 日常用户（含开发者） · 关联: G3
  • 主流程: 进入会话 git-zone 显示分支与四态（干净/已暂存/有改动/冲突） → 暂存/取消暂存 → 提交弹可选 message → 完成刷新
  • 替代: AI 一回合结束后自动刷新；手动操作后刷新
  • 异常: 非 git 仓库（git-zone 隐藏）；冲突时提交被拒（回显）；git 未安装（降级隐藏）
  • 后置: 工作目录 git 状态如实反映；Diff/解决冲突入口触发右抽屉

UC-7: 多窗口/runtime 侧增删会话实时同步

  • Actor: 日常用户 · 关联: G3
  • 主流程: runtime 广播会话列表 → 前端订阅 → 侧边栏列表实时刷新（不重载全量历史）
  • 异常: 订阅中断（下次进入重载）
  • 后置: 侧边栏列表与 runtime 真实状态一致

UC-8: 用户在侧边看到真实文件改动

  • Actor: 日常用户 · 关联: G3
  • 主流程: 侧边文件视图聚合当前会话所有回合文件改动（跨回合合并） → 显示新增/修改/删除 + 行数 + 冲突标注 + 过滤
  • 异常: 无改动（空态）
  • 后置: 文件树反映真实改动，与消息流变更卡片语义对齐

关键语义区分（C12）：「真实 git 状态」（git-zone 用）与「per-turn 改动」（变更卡片/文件视图用）是两条独立数据，各管各的。git-zone 反映工作目录全量状态（含用户手改、IDE 外操作）；per-turn 改动只反映单个 AI 回合。二者并行显示，不互吞。

5. UI/UX 场景（Interface Scenarios）

依据 docs/page-design/v3/panel/spec.md（5 固定 zone）与 draft-companion-zones.html（git-zone 四态）。

页面线框

┌─ Panel ─────────────────────────────┐ │ ① panel-header 会话元信息 │ │ ② message-stream 消息流 + 回合折叠 │ ← F1/F2 流式；F3 重试；F4 排队气泡 │ ③ progress-zone 单会话进度 │ │ ④ composer 输入区+工具区/steer │ ← F4 pending 气泡靠右虚线脉冲 │ ⑤ git-zone 暂存/提交/Diff 入口 │ ← F12 四态展示 └─────────────────────────────────────┘ ┌─ 右抽屉 ─┐ │ Terminal │ ← F7 widget │ Browser │ └──────────┘ ← F8 容器，F12 Diff 按钮触发
  • retry/queue 指示位（F3/F4）：composer 上方独立行（对齐 panel/spec.md:52）
  • git-zone 四态（F12）：干净（绿）/ 已暂存（绿）/ 有改动（橙）/ 冲突（红+danger 竖条）+ 分支 + stats + pill + soft 渐隐底
  • Side Drawer（F8）：从触发 Panel 浮起，固定挂该 Panel，不跨 Panel 覆盖；header 多 tab 容器
  • 文件视图（F10）：聚合所有回合改动成树 + 行数 +N/-N + 冲突标注 + 过滤框

交互流程

  • 发消息（UC-1）：composer 输入 → 发送 → 回合（思考→工具→文本→变更折叠） → 系统通知
  • steer/followup：提交 → composer 上方出 pending 气泡 → 进入 message-stream 后消失
  • Extension 安装（UC-2）：Settings → Extension → 选来源 → 安装 →（候选）选择 → 完成
  • 压缩（UC-3）：slash command → 「压缩中」 → 完成 + 摘要行
  • git 操作（UC-6）：git-zone 显示状态 → 暂存/取消暂存/提交 → 刷新；Diff 按钮触发右抽屉

7. 约束（Constraints）

  • 业务约束：沿用 waves.md §0 的 D1-D10 决策边界（不越界到 tree/工具审批/plugin/附件/搜索）
  • 架构不变：renderer/runtime 分层，命令与推送走统一消息通道
  • mock 与真实门面同构：两侧能力签名一致（mock 必须能验证所有 UI）
  • 错误契约统一：请求级失败走统一错误封装
  • 每 wave 验证基线：类型检查 + 单测全绿 + git 干净后提交
  • 串行执行：不并行派 implementer（会冲突）
  • 技术约束（仅记录不展开）：renderer=Vue3/TS/Pinia/Tailwind；runtime=Node WS；git 读写须防注入
