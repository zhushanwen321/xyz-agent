① 澄清需求

对话流状态撕裂修复 · ① 澄清需求 · business requirements

verdict: pass mode: refactor 6 use cases
TL;DR
  • 把命令式 isStreaming flag 降级为从 messages[] 实体派生的 isGenerating(sid) computed，物理消除"只翻 flag 不收口实体"的撕裂根因
  • 所有异常路径（超时/断连/崩溃/abort/stream_error）统一收敛到唯一出口 finalizeSession(sid, reason)，异常与正常路径走同一收口逻辑
  • agent 忙时用户可追加上下文（B 策略）：键盘 ⏎ 与鼠标发送位都自动路由为 steer，不被静默丢弃或误报错误
  • 不改 pi 协议、不加新 UI 组件，refactor 模式保持正常路径行为等价

业务用例

UC-1 · 正常对话流（baseline，行为等价）

Actor: 用户 · 前置: session idle · 关联: G1（基线，不退化）

主流程：用户输入 → 发送 → assistant 流式回复（text/tool）→ 完成 → 回 idle。后置状态：message.status=complete，toolCall.status=completed，UI 停止按钮消失。

  • AC-1.1 [正常] 单轮对话完成后 isGenerating=false 且 message.status=complete

UC-2 · agent 忙时追加上下文 / 停止重来（B 策略）

Actor: 用户 · 前置: agent 生成中 isActive=true · 关联: G3

追加-键盘：输入 → ⏎ → 自动路由 steer → 入 steering 队列 → 当前回合后投递。追加-鼠标：输入 → 点击发送位（busy 时渲染为 steer 入口）→ 转 steer。停止重来：点停止 → abort → 回 idle → 重新发送。

  • AC-2.1 [正常] 生成中键盘 ⏎ → 走 steer 路径（不触发 prompt RPC）
  • AC-2.2 [正常] 生成中鼠标点发送位 → 转 steer（与键盘对齐）
  • AC-2.3 [正常] 生成中点停止 → abort → isGenerating 变 false + 实体收口
  • AC-2.4 [异常] steer 失败 → isGenerating 不变 + pending 回滚 + toast

UC-3 · 长任务持续生成（>5min）

Actor: agent（glm-5.2 等慢模型）· 前置: agent 正在生成单个长 message · 关联: G2

主流程：流式 delta 持续到达 → UI 持续显示活跃态（停止按钮可见 + 气泡 streaming）。异常：超时兜底触发 → 收口实体（非仅翻 flag）→ UI 准确反映"已结束"。

  • AC-3.1 [正常] 长任务期间 isGenerating 持续 true，停止按钮持续可见
  • AC-3.2 [边界] 超时触发 → 实体收口为 error/end_not_received → isGenerating 派生 false

UC-4 · 异常路径统一收口

Actor: 系统（runtime 崩溃 / WS 断连 / 超时 / abort）· 前置: agent 生成中 · 关联: G1

主流程：异常事件 → finalizeSession → 所有 streaming message→终态 + running toolCall→收口 + pendingSend 清理。后置：isGenerating 派生 false（因实体已终态），无残留 streaming/running。

  • AC-4.1 [异常] runtime 重启 → useConnection 调 finalizeSession → 所有 streaming 实体收口 → isGenerating=false
  • AC-4.2 [异常] 超时 → finalizeSession('timeout') → 实体收口
  • AC-4.3 [异常] abort → 前端乐观清 pendingSend + runtime 广播 message.complete{aborted} 兑底收口实体
  • AC-4.4 [异常] message.stream_error（pi 发 error 不发 agent_end）→ effects 调 finalizeSession('stream_error')
  • AC-4.5 [异常] message.error（pi 真流错误）→ effects 调 finalizeSession('error')

UC-5 · 编辑重发（editAndResend，行为等价基线）

Actor: 用户 · 前置: session idle，存在历史 user 消息 · 关联: G1

主流程：编辑最后一条 user 消息 → truncate（含该消息及后续）→ appendUser 新文本 → send。后置：与 UC-1 send 等价（pendingSend 空窗 + message_start 接管）。

  • AC-5.1 [正常] editAndResend 的 pendingSend 生命周期与 send 对称

UC-6 · landing 态首发（submitFirstMessage）

Actor: 用户 · 前置: 新 session（无 sessionId）· 关联: G1

主流程：输入 → create session → send（create 后有 sessionId，pendingSend.add）。

  • AC-6.1 [正常] landing 首发的空窗期由 pendingSend 覆盖（create 后 add）

约束 & 不做

约束

  • 不改 pi 协议：pi 子进程行为不动，只动 runtime 翻译层和 renderer
  • 行为等价：refactor 模式，现有正常路径行为保持（行为契约见架构 §12）
  • 测试框架 vitest：禁止 node:test（项目规范）
  • 超时可配置：XYZ_STREAMING_TIMEOUT_MS env，默认 24h（D-003）。语义：24h = 用户接受「pi 静默卡死时靠手动点停止」，timer 机制保留但实质不触发；runtime 重启 / WS 断连事件是主要卡死检测手段
  • 超时行为变更（BC）：原 5min 硬编码 → 可配置 24h，且超时从"翻 flag"改为"收口实体"（system-arch BC-3）

不做

  • 不改 pi 的 turn_end / agent_end 事件语义（event-interpreter 的"complete 由 agent_end 独占"设计保留）
  • 不做多 panel 并发流式（G-023 仍 DEFERRED）
  • 不改 session 隔离机制（规则 #7 不动）
  • 不加新 UI 组件（复用现有 Composer / Block / Toast / QueueBubble）；例外：F4 鼠标发送位 busy 时转 steer 是搭便车新交互（模板三态微调，非新组件）
