P0 Issues（阻塞项，必须先做）

#1send.rejected WS 类型契约定义
P 级 P0类型 模块（契约）Blocked by 无推荐 Strong关联 §7 protocol.ts / §8 / D-006
already processing 场景需要一个不污染对话流的独立 WS 类型。当前 message.error 被 5 处复用（hook 拦截/hook 异常/prompt 失败/abort 失败/session 退出），操作拒绝语义与流终止语义混杂。需在 shared/protocol.ts 定义 send.rejected 类型，作为 runtime→renderer 的防御性反馈通道。P0 理由：契约先行——#4 和 #5 都依赖此类型定义。

方案对比

方案 A：新增独立 ServerMessageType.sendRejected 枚举值

改动 protocol.ts 新增枚举值 + SendRejectedPayload（sessionId/reason/message）

优点 语义正交，消费方明确（不进对话流，不翻流式态）；CW gate AC-4 可机器检查

缺点 扩展类型表（但 send.rejected 是终态操作反馈，不会再扩展）

方案 B：复用 message.error 加 kind 字段区分

优点 不扩展类型表

缺点 消费方仍需按 kind 分流，effects handler 逻辑分叉；kind 字段蔓延到所有 message.error 消费方

取舍 方案 A（D-006 已拍板）。派生模型已消除 message.error 的撕裂副作用，send.rejected 的真正价值是 UX 语义分离。放弃 B：kind 字段未解决"消费方逻辑分叉"，且 message.error 进对话流的副作用需每个消费方各自 guard。

验收标准

  • AC-1.1 [正常] protocol.ts 含 ServerMessageType.sendRejected 枚举值 + SendRejectedPayload（sessionId: string, reason: 'busy', message: string）
  • AC-1.2 [正常] sendRejected 不触发 message.* handler（类型独立，effects 注册表无映射）
#2chat.ts 派生模型重构（核心根因修复）
P 级 P0类型 模型Blocked by 无推荐 Strong关联 §4 / §5 / D-005 / D-010 / AC-1 / AC-2
当前 isStreaming 是命令式 ref<boolean>，由 N 条事件路径分别翻转。撕裂根因：每条路径只翻 flag 不收口实体。本 issue 把 isStreaming 从"原始状态"降为"派生态"，物理消除不一致。P0 理由：核心根因修复，#3/#5/#6/#8 全部依赖 chat.ts 的新模型（computed + finalizeSession）。

方案对比

方案 A：per-session computed scan（D-005 已拍板）

改动 删 isStreaming/streamingSessionId/dispatchingSessionId/streamingTimer 4 个 ref + setStreaming/setDispatching/resetActive 3 个 mutation；加 isGenerating(sid)/isActive(sid) computed + pendingSend Set + finalizeSession(sid, reason) action

模型 isGenerating(sid) ≡ ∃ m ∈ messages[sid], m.status='streaming'；scan 限定 messages.value.get(sid)

优点 零手动维护，物理不可能不一致；sealed 不变式（finalizeSession 后实体终态）

缺点 O(n) scan（n<1000 微秒级）

方案 B：增量 Set 维护（isGeneratingSessionIds: Set<string>）

优点 O(1) 查询

缺点 重新引入手动维护一致性——每条新事件路径都要记得"既改实体又改 Set"，原始 bug 类复发；sealed 不变式难以保证

取舍 方案 A（D-005 已拍板）。核心收益是消除全部写路径维护 bug。O(n) 读是可接受的代价（n<1000 微秒级）。放弃 B：增量 Set = 重新引入手动维护 = 原始 bug 复发，复杂度归位失败。

验收标准

  • AC-2.1 [正常]（UC-1） grep -rn "isStreaming" chat.ts 无 ref<boolean> 声明，无 setStreaming 调用
  • AC-2.2 [正常]（UC-1） isGenerating(sid) 为 computed，scan 限定 messages.value.get(sid)（per-session，防跨 session 误伤）
  • AC-2.3 [边界] isActive(sid) = isGenerating(sid) ∨ pendingSend.has(sid)
  • AC-2.4 [异常]（UC-4） finalizeSession 把 streaming message→终态（aborted→complete，D-008 非 error；stream_error/timeout/disconnect/restart→error）+ running toolCall→收口（一律 end_not_received，除 stream_error/error→error；迟到 tool_call_end 覆盖到 completed，D-011）+ pendingSend.delete
  • AC-2.5 [并发] finalizeSession 幂等（重复调用不报错，sealed 后实体不变）。实现约束：行为按终态类分 3 支，reason 原样透传 logger

P1 Issues（核心）

#3chat-message-effects.ts handler 迁移 + sealed guard
P 级 P1类型 模块Blocked by #2推荐 Strong关联 §7 effects / §5 sealed（D-010）/ AC-3
effects 注册表当前有 3 处 setStreaming(false)（message.complete/error/stream_error），迁移为调 finalizeSession。message_start handler 加 pendingSend.delete（接管空窗）。sealed guard：delta 类 handler 检查 last assistant status≠streaming 则 return（防晚到事件污染终态）。
取舍 方案 A（全部终态 handler 改调 finalizeSession + sealed guard）。finalizeSession 是 #2 建立的唯一收口出口，effects 必须统一调用。sealed guard 是 D-010 的实现。放弃 B：维持 N 条路径各自收口正是当前 bug 模式。

验收标准

  • AC-3.1 [正常]（UC-1） message.complete handler 调 finalizeSession（非 setStreaming）
  • AC-3.2 [异常]（UC-4） message.stream_error / message.error handler 调 finalizeSession('stream_error'/'error')
  • AC-3.3 [正常] message_start handler 含 pendingSend.delete(sid)
  • AC-3.4 [边界] finalizeSession 后 text_delta 到达 → handler 检查 status≠streaming → return（幂等丢弃，D-010 sealed）
#4message-dispatcher already processing → send.rejected（runtime 预检）
P 级 P1类型 模块（runtime）Blocked by #1推荐 Strong关联 §7 message-dispatcher / §8（D-009）/ AC-4
message-dispatcher sendPrompt 当前不分类 prompt 失败——忙时调 prompt 被 pi 拒绝后广播 message.error。改为 runtime 预检：sendPrompt 入口检查 activeSession.isGenerating，忙则直接广播 send.rejected 不调 pi.prompt。
取舍 方案 A（D-009 已拍板，runtime 预检）。不依赖 pi 错误字符串匹配（pi 升级改文案不断）；send.rejected 是纯防御（B 策略下前端 busy 走 steer，本不该触发）。放弃 B：依赖 pi 错误文案 "already processing"，pi 升级即断。

验收标准

  • AC-4.1 [正常]（UC-2） sendPrompt 入口检查 isGenerating，忙则广播 send.rejected（不调 pi.prompt）
  • AC-4.2 [异常] grep -rn "already processing" message-dispatcher.ts 无 message.error 广播在 busy 分支
#5useChat.ts 编排重构（B 策略 + pendingSend 迁移）
P 级 P1类型 模块Blocked by #2, #3推荐 Strong关联 §7 useChat / D-001（B 策略）/ BC-2 / BC-5
send/editAndResend/steer/abort 全部迁移：(1) send/editAndResend 用 pendingSend.add/delete 替代 setDispatching（submitFirstMessage 经 send() 复用）；(2) B 策略路由——isActive(sid) 时 send 自动转 steer（D-001）；(3) send.rejected 监听——收到时回滚 pendingSend + toast；(4) abort 乐观清 pendingSend（实体靠 runtime 广播兜底，D-008）。
取舍 方案 A（useChat 内统一路由）。B 策略单一入口，Composer 无需感知路由逻辑。放弃 B（Composer 层路由）：路由逻辑分散到组件层，与"编排归 useChat"分层矛盾；split panel 多 Composer 实例时路由逻辑重复。

验收标准

  • AC-5.2 [正常]（UC-5） editAndResend 的 pendingSend 生命周期与 send 对称
  • AC-5.3 [异常]（UC-2） send.rejected 收到 → pendingSend 回滚 + toast，isGenerating 不变
  • AC-5.4 [正常]（UC-4） abort 乐观清 pendingSend，实体收口靠 runtime 广播兜底
#6useConnection.ts resetActive → finalizeSession 迁移
P 级 P1类型 模块Blocked by #2推荐 Strong关联 §7 useConnection / AC-3
useConnection 的 onRuntimeRestarting/onRuntimeFailed 两处调 resetActive()，迁移为调 finalizeSession('restart'/'disconnect')。resetActive 删除（被 finalizeSession 吞并）。
取舍 方案 A（两处直接改调 finalizeSession）。最小改动，路径不变只换收口函数。放弃 B（useConnection 层自行收口实体）：跨层突变 store 实体违反分层，且每个连接事件 handler 都要重复收口逻辑。

#7Composer.vue B 策略鼠标发送路由（F4）

取舍 方案 A（D-6 已拍板，busy 时发送按钮改 steer 图标）。键盘/鼠标对齐（B 策略完整）；复用现有布局不加新组件。放弃 B（disabled）：与 B 策略矛盾（用户以为"发不了"），键盘 Enter 已走 steer 但鼠标不对齐。

P2 Issues（重要）

#8超时兜底改为收口实体 + 可配置阈值
P 级 P2类型 流程Blocked by #2推荐 Strong关联 §10 D-3（D-003）/ D-4（D-007）/ BC-3 / UC-3
streamingTimer 的 callback 当前调 setStreaming(false)（假收口），#2 删除 setStreaming 后 callback 必须改为调 finalizeSession('timeout')。同时阈值从硬编码 5min 改为可配置（XYZ_STREAMING_TIMEOUT_MS env，默认 24h）。P2 而非 P0 理由：callback 改 finalizeSession 是 #2 删 timer 时的编译必然，实际随 #2 同 Wave。独立标 P2 的部分是"阈值可配置"——增强但非核心根因修复。默认 24h 实质不触发（D-003 拍板），卡死检测主要靠 runtime 重启/WS 断连（#6）+ 用户手动停止。
取舍 方案 A（D-003 + D-007 已拍板）。timer 机制保留（防 pi 静默卡死）；24h = 用户接受手动停止的 UX 妥协。放弃 B（完全删除 timer）：pi 静默卡死（进程活/WS 连/不 emit）时无任何兜底，UI 永久卡死。
