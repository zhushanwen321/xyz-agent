TL;DR · 核心结论
  • refactor 模式：重构 renderer store 内部状态模型 + runtime 预检 + protocol 枚举扩展，不引入新外部依赖/新权限模型/新跨进程边界，整体风险面远小于功能新增型 topic
  • 56 个 issue×维度单元格中无 ❌（不可接受），安全维度全 ✅；风险集中在数据完整性 / 并发（sealed 不变式 + finalizeSession 幂等）
  • 22 条缓解项回灌：18 条代码测试 + 4 条骨架约束 + 1 条反模式 grep + 1 条契约，全部标验收方式
  • 5 项高不确定性副作用（computed 响应式/sealed 边界/幂等/竞态窗口）标记 SV-1~SV-5 进⑤骨架验证，结论回写本节
  • 关键边界：sealed guard 仅对 delta 流类生效，tool_call_end 不 sealed（允许覆盖 end_not_received → completed，否则真实工具结果丢失）
本 topic 是 refactor 模式。所有 issue 方案都是"消除现有 bug / 收口现有散乱逻辑"。瘦身杠杆在"逐 issue × 7 维度"的 ✅ 一行化——✅ 维度一行理由，⚠️ 维度按「副作用 / 缓解 / 残余」展开。

#1 send.rejected 契约 — 方案 A

安全✅数据✅性能✅并发✅稳定性✅
纯类型层声明，零运行时开销，无跨表事务，无并发控制，不影响可用性。
兼容性⚠️
副作用 新增枚举值，灰度共存：新前端+旧runtime 监听永不触发（B 策略下本就不触发）；旧前端+新runtime effects 无映射走 no-op（安全降级，仅 toast 不显示）。缓解 枚举追加非替换，旧前端 no-op 非崩溃。残余 接受——防御兜底本不该触发。
可观测⚠️
副作用 触发即前端路由异常信号，无告警不可见。缓解 runtime 广播时 console.warn + 计数（落盘）。残余 无独立指标，靠日志事后排查。

#2 chat.ts 派生模型（核心） — 方案 A

安全✅兼容✅可观测✅
store 内部重构不跨进程无新边界；外部 API 调用方在 #6 迁移；状态可从实体随时推导。
数据⚠️
副作用 isGenerating 依赖实体正确性，finalizeSession 必须收口所有 streaming（多气泡场景）。缓解 复用 message.complete 的全量收口逻辑；幂等（AC-2.5）。残余 toolCall 收口诚实态与 sealed 交互见 #3（需骨架验证）。
性能⚠️
副作用 computed scan，流式 ~20/s × O(1000) = 2万次/s。缓解 per-session 限定 get(sid) + lazy cached + .some() 短电路。残余 n<1000 微秒级；computed 响应式追踪需⑤骨架验证（SV-1）。
并发⚠️
副作用 finalizeSession 多路径并发触发；isActive check-then-act 空窗。缓解 finalizeSession 幂等 + pendingSend 补全空窗。残余 幂等性需骨架验证（SV-3）。
稳定性⚠️
副作用 pi 静默卡死兜底依赖 timer(24h)+重启/断连+手动停止。缓解 timer 保留双保险。残余 极端组合下 UI 卡最长 24h（D-003 妥协）。

#3 effects 迁移 + sealed guard — 方案 A

安全✅性能✅稳定性✅兼容✅
handler 内部迁移无特权路径；sealed guard O(1)；单一收口减少遗漏路径；message.* 契约不变。
数据⚠️
副作用 sealed guard 误 sealed tool_call_end → 真实工具结果丢失（高严重度）。缓解 sealed 仅对 delta 流类，tool_call_end 不 sealed（允许覆盖）。残余 边界判定需骨架验证（SV-2）+ 测试双重防护。
并发⚠️
副作用 晚到事件与 finalizeSession 竞态，handler 入口 status 检查是决胜点。缓解 happens-before（Vue ref 同步写入无真并行）。残余 时序正确性需骨架验证。
可观测⚠️
副作用 sealed 丢弃是 silent drop。缓解 debug 级日志落盘。残余 prod 需 XYZ_LOG_LEVEL=debug，dev 排查足够。

#4 dispatcher 预检 — 方案 A

安全✅性能✅稳定性✅兼容✅
预检基于 runtime 内部态，message 字段固定文案；O(1) 检查；避免无效 RPC；runtime 内部行为。
数据⚠️
副作用 runtime/renderer 两套 isGenerating 可能不一致导致误判。缓解 runtime 在 broadcast 终态前同步置 false；误判不破坏对话流（仅多/少 toast）。残余 既存架构事实，本 topic 不修。
并发⚠️
副作用 split panel 同 session 双击在 hooks await 期间都过预检。缓解 预检置最入口；pi 是最终单写者兜底。决断（D-014） catch 一律 message.error（不分类），send.rejected 只走预检。残余 极端场景，pi 兜底，严重性低。
可观测⚠️
副作用 预检拦截与 pi 拒绝两路径都广播 send.rejected 需区分。缓解 runtime 预检 log [dispatcher] preemptive reject；catch log [dispatcher] prompt failed。残余 无独立指标，靠日志。

#5 useChat 编排 — 方案 A

安全✅性能✅稳定性✅可观测✅
composable 编排无特权；Set O(1)；编排归一；派生态可推导。
数据⚠️
副作用 pendingSend 跨多路径生命周期。缓解 Set delete 天然幂等 + per-session 隔离。残余 无实质残余（需骨架验证 SV-5 关联）。
并发⚠️
副作用 isActive check-then-act，send 与 abort 并发竞态。缓解 isActive 派生态保证最终一致；实体靠 runtime 广播兜底。残余 check-then-act 窗口正确性需骨架验证（SV-5）。
兼容⚠️
副作用 B 策略行为变更（BC-2），用户心智需迁移。缓解 键盘已走 steer，#7 鼠标对齐；editAndResend 保留 guard（BC-5）。残余 接受——D-001 拍板，QueueBubble 反馈链完备。

#6 useConnection 迁移 — 方案 A

安全✅性能✅稳定性✅兼容✅可观测✅
连接事件 handler 迁移无新边界；O(messages[sid])；统一收口；内部调用迁移无外部消费者；reason 可 log。
数据⚠️
副作用 resetActive 只清 flag 不碰实体 → finalizeSession 收口实体（行为变更=修复 G1）。缓解 reason 映射 error/end_not_received符合 §5。残余 无（方向正确）。
并发⚠️
副作用 重启/失败时可能有在途 message.* 帧。缓解 finalizeSession 幂等 + sealed guard。残余 依赖 #2 幂等，无新增。

#7 Composer 鼠标 B 策略 — 方案 A

安全✅数据✅性能✅并发✅稳定性✅可观测✅
UI 模板调整无特权；不直接操作实体；computed 无额外开销；并发由 useChat 处理；复用现有布局；现有状态反映。
兼容⚠️
副作用 发送位三态重构，stop 与 steer 布局可能重叠/错位。缓解 UI 回归测试（BC-1）+ 复用现有布局 + 三态 testid 覆盖。残余 像素级一致性需人工验证。

#8 超时兜底收口 — 方案 A

安全✅性能✅
timer callback 逻辑变更无新边界；XYZ_ 前缀自动过白名单；单 timer 无性能影响。
数据⚠️
副作用 timer 与 message.complete 并发触发 finalizeSession。缓解 finalizeSession 幂等 + clearTimeout 防重复。残余 依赖 #2 幂等。
并发⚠️
副作用 timer×complete×断连×重启多路径收口竞态。缓解 幂等 + sealed guard。合并到 #2 并发风险。残余 无（幂等保证最终一致）。
稳定性⚠️
副作用 阈值 5min→24h，UI 卡窗口扩大。缓解 timer 保留 + 重启/断连为主检测 + 手动停止可用 + env 可调。残余 极端组合卡 24h（D-003 妥协）。
兼容⚠️
副作用 BC-3 行为变更：翻 flag→收口实体（修复）+ 阈值变更。缓解 env 向后兼容默认 24h。残余 升级用户若依赖原 5min 需手动设 env。
可观测⚠️
副作用 24h 触发是极端异常信号。缓解 callback warn 级日志落盘。残余 极罕见，日志做事后归因。

需⑤骨架验证的副作用（SV-1~SV-5）

SV-1 · #2 性能（M5/M6）

computed scan 响应式追踪 + 缓存命中

验证 isGenerating(sid) 基于 messages.value.get(sid)?.some(...) 在数组替换时正确失效、多处引用共享缓存、per-session 隔离。预期 Vue 3 reactivity 成熟，n=1000 <1ms。stub 进⑤骨架。

SV-2 · #3 数据完整性（M8/M10）

sealed guard 边界（delta sealed vs tool_call_end 覆盖）

验证 sealed 对 text_delta/thinking/tool_call_update 生效，对 tool_call_end 不生效（允许覆盖 end_not_received→completed）。delta 类增量追加 sealed 正确；tool_call_end 终态确认覆盖诚实态正确。

SV-3 · #2/#8 并发（M4/M7/M20）

finalizeSession 幂等 + 多路径并发收口

验证 timer/WS 断连/重启/迟到 complete 多路径触发时幂等：重复调用不报错、sealed 后 no-op、clearTimeout 生效。单线程 JS 无真并行，竞态本质是事件顺序。

SV-4 · #4 并发（M12/M13）已决断

runtime 预检竞态窗口 + catch 路由

D-014 决断：catch 一律 message.error（不分类）。send.rejected 只由预检触发。catch 不区分 pi 拒绝 vs 其他——D-009 禁字符串匹配 + 无可靠结构化判据。NFR 自承保守 fallback 显式排除。

SV-5 · #5 并发（M15）

isActive 路由 check-then-act

验证 useChat.send 检查 isActive=false → pendingSend.add → 并发 abort 乐观清的路由正确性。isActive 派生态保证最终一致，实体靠 runtime 广播兜底。check-then-act 窗口瞬态不一致但不破坏最终收口。
