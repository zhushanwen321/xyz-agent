constraints · 并行约束 / 裁决

并行约束与 Prefactor 裁决

并行约束

  • Wave 间严格串行（DAG W1→W2→W3→W4），无 Wave 间并行。「并行组」列指 Wave 内部组件文件并行（Wave 2 的 #5/#6）
  • 同一并行组内最多 3 个 subagent 并行（Semaphore 限制）
  • 同一文件不允许多 Wave 同时修改；被多 Wave 渐进扩展的文件（useNewTaskFlow.ts / api/domains/git.ts / runtime git-service.ts）都是 串行依赖，非并行
  • 前端 Wave 需对应后端 API 同 Wave 内先就绪（Wave 3 CreateBranchModal ← runtime createBranch；Wave 2 BranchSelectPopover ← runtime checkout）

Prefactor Wave 裁决

无 Prefactor Wave。⑤§7 现有代码映射明确「无 delete/split 现有代码」（新建任务是增量功能）。

所有 merge 项（session.create 扩签名、GitCommand 加 checkout、protocol 加消息）都是 功能 Wave 内部的实现动作，不是可独立提前的重构。

refactor 场景的「行为等价测试约束」随各 Wave 的回归用例落地（如 Wave 1 含「既有 create() 无参调用仍工作」回归）。

职责

读「测试验收清单」全量 → 跑测试套件 → 核对每条用例 ID 的 PASS/FAIL/缺失 → 输出覆盖率报告。设计→实现的闭环闸门。

执行流
  1. read execution-plan.md「测试验收清单」（全量用例 ID + 断言摘要 + 归属 Wave）
  2. 跑测试套件（全量）
  3. 把每条 PASS/FAIL/缺失映射回清单用例 ID（按断言摘要核对，非按文件名）
  4. 清单状态列填 PASS / FAIL / 未实现 / [DEVIATED]原因
  5. 输出覆盖率报告：清单用例 PASS 数 / 总数 + 未过用例明细
验收标准
  • 测试验收清单全量用例 PASS（任一 FAIL / 未实现 = 整个实现未完成，回对应功能 Wave 补）
  • 无 [DEVIATED] 未经用户确认（偏离需登记原因 + 用户拍板 + 判断是否回流⑤改设计）
  • 覆盖率报告输出（清单 PASS 数 / 总数）
