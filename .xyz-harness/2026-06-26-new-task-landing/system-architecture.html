核心结论 (TL;DR)
  • 核心计算 = 前端技术编排（5 步 UI 状态机 + IPC 调度 + git 读取 + session 生命周期编排），非领域规则编排。现有 runtime 三层是稳定基座，不新增后端分层。
  • NewTaskFlowState 显式状态机（8 态 + ~17 转换）管 overlay 互斥嵌套，唯一不可逆终态 = completed；cancelled 非终态（空 session 可重选复活）。
  • 复用为主、扩展为辅：cwd 透传 runtime 已支持（前端补全 T1）；GitService.createBranch 扩 port 能力集（UC-6）；GitService.getStatus 复用接入 dirty。
  • T2/T3 双双打回：git-info 与 GitService 职责正交不合并（T2 红队终裁）；RecentWorkspace 从独立 LRU 缓存降为派生函数 recentWorkspaces(sessions)（T3，伪边界）。
  • T3 交叉验证打回：RecentWorkspace 从独立 LRU 缓存降为派生函数 recentWorkspaces(sessions)（独立缓存是伪边界）。
  • 12 条行为契约（BC-1~BC-12）界定「新建任务功能本体」与「搭便车 ticket」边界，保证重构行为等价。BC-12 = UC-7 非 git 目录 branch chip 隐藏既有行为保持。

§2 设计立场 Design Stance

核心计算 = 技术流程编排，非领域规则编排

立场

三层够，不套 DDD 四层

核心是 UI 状态机 + IPC 编排，无复杂领域规则引擎。现有 Transport→Service→Adapter 已捕获「协议变化 vs 业务编排 vs 外部适配」的不对称。加四层是零价值空壳层。

稳定基座

runtime 三层不新增

现有 runtime 是稳定基座。「新建任务」复杂度集中在前端编排层（状态机 + overlay 嵌套），后端是复用现有 session/git 服务 + 少量扩展。

复杂度归位

UI 编排不污染 runtime

cwd 透传 runtime 已具备，前端补全；dirty 接入复用 GitService.getStatus。前端编排在 renderer，不进 Service 层。

§4 核心模型 Core Model

模型、不变式与建模理由

模型类型不变式建模理由

§5 状态流转 State Flow

NewTaskFlowState 状态机（见顶部主角图②）

Status 枚举

8 个阶段（只描述阶段）

idle | landing | dir-popover | branch-popover | dir-dialog | branch-modal | completed | cancelled

完整转换表见顶部主角图②。

终态

唯一不可逆终态 = completed

cancelled 非终态（建模G-4）：空 session 永久保留可被重选 → NewTaskFlow 复活（cancelled → landing）。真正终态只有 completed。

失败处理

不引入 Reason 字段

主流程发送失败 → 折叠为 landing 的 composer 子态（flow 状态不变）。overlay 内失败 → 停留当前 overlay。两种失败都不改 NewTaskFlowState，故无需 Reason。

不变式守卫

overlay 互斥 + 深模态来源约束

  • 四态至多 1 个 active（spec §4）
  • dir-dialog 只从 dir-popover 进；branch-modal 只从 branch-popover 进
  • landing 是唯一无 overlay 可交互态
  • 非 git 目录约束（UC-7）：gitInfo == null 时 branch-popover/branch-modal 不可达、branch chip 隐藏（实际只走 idle ↔ landing ↔ dir-popover ↔ dir-dialog 子集）

Session 生命周期（已有，复用）

Session 状态机不变（active | idle，由 isGenerating 派生）。新建任务引入的业务里程碑：

create

触发新建即创建空 session（绑定 cwd），status=idle（isGenerating=false）。

首条消息

empty→chatting 业务里程碑，session 进入对话流，status 转 active（isGenerating=true）。落地空态判据 = messageCount===0 && !isGenerating（前端派生，非 status 字段）。

§10 挑战与决策 Decisions

6 个关键决策记录

D-1

核心计算定位为「技术编排」而非「领域规则」

张力：是否需要 DDD 四层？
决策：三层够，不套四层。
理由：核心是 UI 状态机 + IPC 编排，无复杂领域规则引擎。加四层是零价值空壳层（层边界代价与捕获的不称不匹配）。

D-2

RecentWorkspace 建 DTO 不建 aggregate（Q1=A）

张力：未来是否演化成「项目」概念？
决策：DTO，LRU 10 缓存。
理由：task=session 1:1 已定，spec.md 无项目管理意图。DTO→aggregate 升级可逆（加字段），aggregate→DTO 降级难。YAGNI。

D-3

落地空态判据用 messageCount 派生，不引入 empty status

张力：新建空 session 与历史 idle session 如何区分？
决策：messageCount === 0 && !isGenerating 派生，不改 SessionStatus。
理由：SessionStatus 描述「pi 是否在生成」，与「有无消息」正交。强行加 empty 会把两个正交维度揉进一个枚举（状态机不纯）。

D-4

NewTaskFlowState 显式转换表（Q3=A）

张力：5 步 overlay 嵌套用显式状态机还是松散 ref？
决策：显式转换表，非法转换抛错。
理由：spec §4 边缘态「Esc 优先级/overlay 互斥」需代码级保证。松散 ref 在 popover+modal 嵌套时易出 bug。成本不高（8 态 + ~17 转换）。

D-5 打回

git 服务维持分离，不合并（T2）

张力：git-info 与 GitService 是否合并？
决策（审查红队终裁，从候选降为打回）：维持分离，v1 不补 port。
理由：证伪三连「删 git-info」→复杂度仍分散（轻量高频读 vs 重操作是两个变化轴）→模块边界真实，合并是制造耦合。证据强度与 T3 交叉验证相当，应同处置（T3 打回则 T2 亦打回）。dirty 接入 popover 仅复用 GitService.getStatus，不动 git-info。
⚠️ 层边界补充（结构G-3）：deletion test 只判「模块边界」，不涵盖「层边界」——git-info 在 services 层裸 execSync 是独立的分层泄漏（§6）。此分层债修复价值独立于合并决策，是否补 IGitExecutor port 覆盖读路径留 ⑤code-arch 评估。

D-6 打回

RecentWorkspace 从独立缓存降为派生函数（T3）

张力：是否独立 LRU 缓存 + 是否抽象泛型？
决策（追踪后修正）：打回 T3，从独立缓存降为派生函数，不抽象泛型。
理由：交叉验证（建模G-2/演进G-2 两组独立命中）：SessionSummary 已含 cwd+lastActiveAt，distinct cwd top10 可派生，独立缓存是伪边界。泛型抽象同理 YAGNI。说明：Step 1 用户意向=T3 做，但独立追踪的交叉验证推翻（loop 机制设计的价值——独立 subagent 发现主 agent+用户的确认偏误）。Step 6 红队复核此降级。
