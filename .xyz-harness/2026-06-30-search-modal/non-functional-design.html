详细分析

#1 匹配引擎提取为纯函数模块 全部 ✓ · 纯函数无副作用

方案 A（双函数）

✓ 安全 / 数据 / 并发 / 稳定性 / 兼容性 / 可观测

纯函数无 I/O、无外部依赖、无状态。AC-1.2 已强制无副作用（grep ref/reactive/api/transport 无输出）。

✓ 性能

子串匹配是线性 indexOf（非 O(text×q) 嵌套），红队审查 AH-B5 已降级（⌘K 输入手敲无粘贴路径，200+ 字符近乎不可达）。

#2 命令注册表 全部 ✓ · 内存数据结构扩展

方案 A（扩 command store + composable）

✓ 安全

应用命令 action 是编译期静态注册（非运行时外部输入），无注入面；pi slash 命令经 runtime getCommands 透传，已是现有受控边界。

✓ 数据 / 并发

D-016 两区物理隔离（appCommands 静态 ref + slashCommands per-session Map），AC-2.2 已守 session 切换不重算应用命令区。无跨表事务、无持久化。

✓ 性能 / 稳定性 / 兼容性 / 可观测

应用命令 <20 项静态，slash 命令 per-session 小集合，computed 合并开销可忽略。复用现有 store 基建，无新增故障面；扩展不破坏现有 SessionCommand 消费者（applyCommands/clearCommands 不变）。

#3 recents composable 4 项 ⚠️ · localStorage 边界

方案 A（localStorage 持久化）

✓ 安全

Electron 桌面应用（非浏览器），无公网 XSS 注入面（content 非用户可控 HTML，渲染进程加载本地 bundle）；recents 内容是用户自己的文件路径/会话标题（非跨用户数据）。

! 数据一致性影响

事务边界：单次 write 是 localStorage 单 key 整体写入（JSON.stringify 整个 recents 对象），原子性由 setItem 保证（非跨 key 事务，无部分写入）。

并发场景：单实例桌面应用无多用户并发；同进程内 useRecents 是单例 composable，无并发写。

迁移方案：首次使用（无 recents key）→ read() 返回空数组（AC-3.3），write 时创建 key。无存量迁移。

回滚策略：recents 是偏好数据非业务数据，损坏/丢失可接受（最坏清空 key 重建）。

! 并发控制（追踪 G-3.1 修订）

竞态场景：同毫秒连续 write——Date.now() 精度不足致 FIFO 排序不确定。

幂等策略：AC-3.5 同 key 重复确认更新 timestamp 而非新增（幂等）。

锁策略：无锁（单实例单线程）；AC-3.6 timestamp 计数器兜底（Math.max(stored)+1）而非裸 Date.now()，是并发竞态缓解。

分布式考虑：N/A——单实例桌面应用。

! 稳定性影响（追踪 G-3.2 修订）

故障场景：localStorage 配额满 / JSON.parse 失败（脏数据/格式升级）。

降级方案：read() try/catch JSON.parse 失败→空数组；write() 配额满 catch 降级（内存态保留本次写入，不回滚——见 MR-3.3，本次会话内仍显示成功，reload 后丢失，符合「偏好数据可丢失」容忍度）。

熔断/限流：N/A。重试策略：不自动重试（偏好数据，下次 write 覆盖）。SLA 影响：无。

! 兼容性影响

API 变更：无（纯前端 localStorage）。

数据兼容：RecentEntry 结构变更（未来加字段）须处理存量旧格式——read() try/catch JSON.parse 失败降级空数组（MR-3.1）。

客户端影响：recents 跨 reload 保留（D-007 跨会话保留），跨应用版本升级须向后兼容旧 RecentEntry 格式。

灰度/回滚：桌面应用本地无灰度；回滚时旧 key 残留无害（不被读）。

key 命名（G-3.4 修订）：recents key 用 xyz-agent:search-recents（对齐现有 xyz-agent:system-settings 冒号命名约定）。

✓ 性能 / 可观测

localStorage 同步 API 但 recents <20 项小数据，序列化开销可忽略。无关键操作需日志（recents 是偏好数据）。

#4 search real domain 5 项 ⚠️ · runtime WS 边界 + F-1 挂死漏洞

方案 A（单 domain + allSettled + 分组）

✓ 安全

query 是用户输入的搜索词，进 matchFilter 子串匹配（indexOf，非 eval/SQL），无注入面。

! 数据一致性影响（追踪 K-1 补充）

事务边界：domain 是只读查询编排，无写入（recents 写入归 #3/#6）；allSettled 不保证跨源一致性（各源独立 resolve）。

K-1 · 关键约束error 冒泡链（AC-4.5）是数据完整性不变式——domain 编排 file 源必须直调 composer.getFileCandidates（经 pending reject 透传 error envelope），不经 useFileSearch.load（其在 useFileSearch.ts:39-43 静默 catch 吞错降级空数组）。若经吞错层，file 源失败永不冒泡到 domain catch，AC-8.2「不静默」假性 PASS。

迁移/回滚：N/A（只读）。

! 性能影响

预期负载：单用户查询，非高并发。但每次按键（无 debounce 时）触发 allSettled 并行查 3 源（命令/文件/会话）——file.search 是 runtime 全量递归（深度 8 / 上限 5000），session.list 是全量跨项目扫描。

关键路径延迟：file.search 全量递归是大仓库主延迟源；P99 目标单次查询 <500ms（用户感知流畅阈值）。

扩展性瓶颈：大仓库（>5000 文件）触发 MAX_SEARCH_RESULTS 截断（file-service.ts:59，D-021 已反哺 5000）；多项目会话库增长时 session.list 扫描耗时线性增长。

优化方案：① debounce(120ms) 已按 D-020 提前到 #7；② file 源复用 useFileSearch 的 session 级缓存（AC-4.9）；③ matchFilter 前端过滤（全量候选拉取后本地子串匹配）。

! 并发控制

竞态场景：快速连续查询——旧响应晚到覆盖新结果（check-then-act：loadSeq++ → await → seq===loadSeq 才写）。

幂等策略：BC-9 loadSeq 序列号守卫——domain query() 内部维护自增 seq，await 后比对，旧响应丢弃。

锁策略：无锁（单用户单实例，loadSeq 足够）；loadSeq 迁移正确性是关键不变式（D-022 + MR-4.1）。

! 稳定性影响

故障场景：① runtime WS 断连——file.search/session.list 永不 resolve（pending）或 reject；② session 不存在（AC-4.8）——file 源无 sessionId 取 cwd + slash 源分区空。

降级方案：allSettled 容错——单源失败（reject）不阻塞其他源，失败源返回空 section（AC-4.8）。

F-1 · 追踪关键漏洞（D-023 新建 #17）ws-client.ts onclose 不 reject in-flight pending（pending.ts 无 clear/flush），file/session 源的 pending永远不 settle → Promise.allSettled 只在所有输入 settled 后才 resolve → domain query() 永久 await → 浮层永久 loading 挂死，无 toast 无降级。MR-4.2 测试只 mock 立即 reject，掩盖了这条永不 settle 路径。修复：新建 #17 [P1]，domain query() 对 WS 源加超时 race（10s 量级，对齐 runtime），超时→reject→allSettled settle→分组空态+toast。根因修复。

熔断/限流：debounce(120ms) 是客户端侧软限流。重试策略：不自动重试（用户每次按键是新查询）。SLA：无可用性目标（桌面应用本地运行）；WS 断连时浮层降级运行（命令源仍工作，因命令注册表是内存）。

! 可观测性

日志：domain catch 错误须 toast 反馈用户（AC-8.2，no-silent-catch lint 强制）；但 allSettled 单源 reject 的静默降级（AC-4.8 转空 section）是预期行为，不应 toast 噪音——须区分「单源失败降级」（静默，分组空态）vs「全源失败/跳转失败」（toast）。

指标/追踪/告警/审计：无业务指标/跨服务追踪/告警需求（单进程内调用）；recents 写入是用户行为日志（D-007 明确），无需额外审计。

! 缓存失效竞态（追踪 K-3 补充）

竞态场景：AC-4.9 要求复用 useFileSearch 的 session 级缓存，但该缓存失效靠 useFileSearch.setupInvalidation 驱动（watch chatStore fileChanges → invalidate），目前由 CommandPopover onMounted 绑定。新 search domain 消费缓存时谁绑失效 watch 未定义——若 CommandPopover 未挂载，agent 改文件不触发 invalidate → stale cache，用户搜不到刚改的文件。

缓解：补 AC-4.10（MR-4.4）——search domain 消费缓存时须自绑 setupInvalidation watch（不依赖 CommandPopover 挂载），或在 query() 调用前校验缓存新鲜度。

#6 跳转编排 2 项 ⚠️ · 稳定性/可观测 + GAP-2 吞错层

方案 A（switch 分发）

✓ 安全

跳转目标是 SearchItem（来自受控数据源），非用户自由输入 URL/路径；file.read 走 runtime 受控路径校验。

✓ 数据

recents 写入归 #3 的数据一致性（单 key 原子写）；session.switch 是现有受控操作。部分失败容忍（GAP-1）：跳转成功 + recents 写入失败（localStorage 配额满）是部分失败——MR-3.3 已定义降级（内存态保留，reload 后丢失），符合「recents 是偏好数据可丢失」容忍度，不阻断跳转成功路径。

✓ 性能 / 并发 / 兼容性

跳转是单次用户操作，无性能/并发面；复用现有 useDetailPane/selectSession 不破坏现有消费者。

! 稳定性影响

故障场景：跳转目标失效——① file.read 失败（文件被删/权限）；② session.switch 失败（session 文件损坏/失效，注意 pi 延迟写入见 AGENTS.md 规则#6——但 session.switch 是进程激活层非文件层，正交）；③ 应用命令 action 抛错。

GAP-2 · 假性 PASS 漏洞（D-024 补 AC-6.9）与 #4 AH-E1/E2 同构——file 跳转调 useDetailPane.openPreview，但后者 try/catch 吞错（设 status='error' 不抛出）。file.read 失败时 useSearchJump 的 catch 永不触发，AC-6.5「file.read 失败→toast」假性 PASS。修复：#6 补 AC-6.9——file 跳转须直调 fileApi.read 校验不经吞错层，与 #4 AC-4.5 error 冒泡链对称。

降级方案：三类跳转失败均 toast 错误反馈（AC-6.5/6.6/6.8 + 新 AC-6.9），不静默失败。关键：AC-6.7 异常恢复——跳转先 await 成功再关浮层，失败保持打开让用户重选。

熔断/限流：N/A（用户主动单次操作）。重试策略：不自动重试；session.switch 失败时 AC-6.6 刷新会话列表（消除失效 session）。

! 可观测性

日志：跳转失败 toast 是用户可见反馈（AC-6.5/6.6/6.8/6.9）；但跳转成功路径无日志（成功是默认期望，无需日志噪音）。指标/追踪/告警：无。审计：跳转后写 recents（AC-6.4）是用户行为记录，无需额外审计。

#7 SearchModal 改造 2 项 ⚠️ · 并发竞态 + ⌘K toggle 变更

方案 A（渐进改造）

✓ 安全 / 数据 / 性能 / 稳定性 / 可观测

UI 组件改造，BC 清单逐条验收保证行为等价（AC-7.1~7.9）；渐进改造降低破坏既有交互风险；行数上限 AC-7.12（script ≤300 / template ≤400）。debounce(120ms) 是性能护栏（D-020 提前到 #7）；改造后 LOC 收敛。

! 并发控制

竞态场景：浮层快速 open/close 交替——① open 触发 loadResults（watch open）+ query 变化触发 loadResults（watch query），两者并发；② close 时 query 清空（BC-11）与 pending loadResults 的竞态；③ debounce 定时器残留（close 后仍 fire）。

G1 · close 触发孤儿查询现状 watch(open) close 分支执行 query.value=''（SearchModal.vue:184）→ 立即触发 watch(query)（:180）→ loadResults('') 再发一次 runtime 全量查询（浮层已关）。loadSeq 能丢弃结果（无害），但查询本身不被阻止——debounce 后 AC-7.14 验收可能假性 PASS。缓解：MR-7.1 补充——close 时不仅 clearTimeout 还须守卫该孤儿查询（open flag 检查，close 后 loadResults 直接 return）。

幂等策略：AC-7.14——loadSeq 守卫已覆盖结果竞态；debounce 的 setTimeout 须在 close 时 clearTimeout（与 AC-8.4 loading setTimeout 清理同模式）+ 孤儿查询守卫。锁策略：无锁（loadSeq + clearTimeout + open flag 守卫足够）。

! 兼容性影响（追踪 G2 修订）

⌘K toggle 变更（G2）：AC-7.1 是 [等价/变更]（=true→toggle，源码核验 Sidebar.vue:236 确为非 toggle）——「再按⌘K 关闭」是行为变更非保持。兼容性维度须识别用户肌肉记忆变化（连按⌘K 从无反应→toggle）。本表 §#10 已承认此变更（AH-C5），#7 改造时 AC-7.1 须显式实现 toggle 逻辑（与 AC-10.1 协同）。

BC 清单等价性：其余 BC-2~BC-12 保持项（↑↓导航/高亮/空结果态/图标/loadSeq/鼠标/生命周期/边缘不变式）逐条验收保证等价。

#8 loading 态 + error 态 1 项 ⚠️ · toast 错误文本脱敏

方案 A（延迟显示 + toast）

! 安全影响（追踪 GAP-BL-2 补充）

风险：no-silent-catch lint 强制 catch，但 routeInbound（useConnection.ts:46）原样透传 runtime payload.message → Error.message → toast。message 可能含绝对路径（用户目录结构泄漏）+ 内部 code（out_of_cwd/permission_denied/TIMEOUT/AUTH_ERROR，见 shared/errors.ts）。现有 codebase（useDetailPane/useGitStatus）均无脱敏约定。

影响范围：toast 错误反馈文本（用户可见）。

#17 WS 源超时 race 新建 [P1] · F-1 根因修复 · blocked_by #4

新建 issue · MR-17.1 落地
来源：④非功能设计追踪发现 F-1 漏洞（D-023）。ws-client.ts onclose 不 reject in-flight pending（pending.ts 无 clear/flush），致 domain query() 对 file/session WS 源的 Promise.allSettled 永久 await，浮层永久 loading 挂死，无 toast 无降级。这是 transport 层遗漏的根因修复。

! 安全 / 数据 / 性能 / 并发 / 稳定性 / 可观测（实现期）

修复方案：domain query() 对 file/session WS 源加超时 race（10s 量级，对齐 runtime），超时→reject→allSettled settle→分组空态+toast。

  • 安全 ⚠️：超时 reject 透传的 error envelope 可能含路径/code（与 #8 GAP-BL-2 同类，归残余风险）。
  • 数据 ⚠️：超时后该源降级空 section，结果不完整（预期降级行为，AC-4.8 范畴）。
  • 性能 ⚠️：超时 10s 是延迟代价（最坏用户感知 10s 后才 toast），对齐 runtime 超时量级。
  • 并发 ⚠️：race 实现涉及 timer 与 pending 的并发控制，须 clearTimeout 防泄漏（与 AC-8.4 同模式）。
  • 稳定性 ⚠️：本质是稳定性提升（消除挂死），但实现期是风险点（race 逻辑错误会引入新竞态）。
  • 可观测 ⚠️：超时须 toast 反馈用户（AC-8.2 no-silent-catch）。

✓ 兼容性

纯 domain 内部加超时 race，不改变 WS API 契约，不影响现有 runtime 消费者。
