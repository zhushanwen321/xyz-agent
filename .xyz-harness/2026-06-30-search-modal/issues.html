#1 · 匹配引擎提取为纯函数模块

P0 类型: 模块 blocked_by: 无 推荐强度: Strong

当前 SearchModal.vue:141-155 的 segments()（子串切分驱动 <mark> 高亮）是组件私有函数。架构 §7 要求提取为独立纯函数模块 lib/match-engine.ts——因有两个消费者：① SearchModal 模板渲染（高亮段）；② search real domain 的 matchFilter（前端过滤候选）。共享同一子串核心算法，提取避免逻辑重复。P0（基础设施先行 D-017）：无前置依赖，被 #4 + #7 依赖。

方案对比

A双函数分离（matchFilter + segments）✓ 已选
改动新建 lib/match-engine.ts，导出两纯函数；matchFilter 在 domain 调用（过滤），segments 在 SearchModal 模板调用（高亮渲染）
优点职责单一；两消费者各取所需无耦合；符合 §7「ME 不含分组、不调 api、无副作用」
缺点两个导出，调用方需知道用哪个
适用过滤与渲染是两个独立关注点（本例正是）
B单函数合一（match 返回带高亮段的过滤结果）
改动单一 match(items,q) 返回「过滤后项 + 每项附高亮段」复合数组
优点单一入口，调用方简单
缺点强耦合过滤与渲染——domain 只需过滤却被强制算高亮段（性能浪费）；返回形态不符 SearchItem DTO
适用过滤与渲染总是同时发生（本例不是）
C内联不提取（保持 SearchModal 私有）
改动无新模块，segments 留 SearchModal，domain 另写过滤
优点零改动成本
缺点逻辑重复违反 DRY；§7 模块设计落空
适用只有一个消费者（本例有两个）
取舍决策 · 选择方案 A

过滤与渲染是两个正交关注点——domain 只过滤（matchFilter），SearchModal 只渲染（segments）。职责单一、无强耦合、符合 §7 边界。放弃 B：强耦合违反单一职责；放弃 C：逻辑重复违反 DRY。

验收标准

  • AC-1.1正常grep -n "export function matchFilter\|export function segments" lib/match-engine.ts 两导出均存在 （trace: architecture §11 AC-4）
  • AC-1.2纯函数grep "ref\|reactive\|import.*api\|import.*transport" 无输出（无副作用、无外部依赖）
  • AC-1.3边界空查询 segments(text,"") 返回单元素数组（原文 + hit=false）；matchFilter(items,"") 返回全部 （trace: UC-1 AC-1.2）
  • AC-1.4等价提取后高亮渲染与提取前行为等价（BC-4 保持）
  • AC-1.5边界极大查询串（>200 字符）早退，避免 O(text×q) 拖慢渲染 （AH-B5）
