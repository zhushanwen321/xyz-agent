## 过度设计发现

### 1. 符号分组死占位行（F8 / UC-5 / D-002）— agent-opinionated，实现方式过度

**对象：** D-002「符号类占位策略：UI 保留分组位但结果区显示降级提示（非隐藏整个分组）」，`confirmed_by=agent-opinionated`。产物为 F8 / UC-5 / AC-5.1/5.2。

**Deletion test：** 删掉 F8 → 四类结构变三类（命令/文件/会话），用户少扫一个永远「暂不可用」的分组行。验证其「活性」：
- UC-5 后置状态 =「无（占位，无副作用）」——**无任何状态变更**。
- AC-5.2 =「占位提示不随查询变化（恒定提示文案）」——**不消费查询、不产生结果、不参与匹配**。
- 即 F8 是纯展示性、恒定、零数据消费、零真实交互的**死代码行**。

**为何过度（反方质询）：**
- 一个永远不可用的分组，对用户**零正向价值**，每次唤起浮层都需扫过一行无功能项，是认知噪音；「暂不可用」提示甚至可能被误读为 bug/配置缺失。
- 「后续补符号数据无需改 UI 结构」是典型 **YAGNI 反论证**——为假想的未来保留结构；而将来真接入 LSP 时，加一个 case 分支渲染符号分组的成本极低，现在为此永久占位的收益近乎零。
- 与项目自身的克制内核自相矛盾：D-003（不引 ripgrep，「收益不匹配」）、D-001（符号成本远超三类）都以「克制、收益匹配」为内核，而一个零功能死占位行恰违背同一原则。

**关键边界（纪律自检）：** D-001「四类范围全保留」是 `confirmed_by=ask_user`，范围决策尊重，不质疑。质疑的是 D-002 的**实现方式**（渲染死占位行 vs 隐式预留），而 D-002 为 `agent-opinionated`，按红队纪律可正常质询。

**建议降级方案：** 四类**结构完整性**在 spec / 代码 case 分支层面预留第四类位置（forward-compatible），但 UI 上**不渲染死占位行**——数据源就绪时该分组才出现。具体：
- spec.md 标注「符号分组待 LSP 接入后启用，UI 暂不渲染占位」。
- requirements.md：F8 由「符号占位降级提示」改为「符号分组结构预留（UI 不渲染，待 LSP）」；删除 UC-5（或降级为 spec 层注释）；删除 AC-5.1/5.2（无对应死占位可验）。
- 保留 D-001 四类范围、D-002 降级语义不变，仅改实现载体。
