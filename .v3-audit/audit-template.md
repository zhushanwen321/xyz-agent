# v3 审查结果模板（阶段 A/B 通用）

> 每个审查 wave 产出一份 `wave-<编号>-<区域>.md`，遵守本模板。
> 目标：可追溯（设计+实现双证据）、可聚合（根因归类）、三态+多余态判定。

## 一、Wave 汇总表（每条目一行，快速扫描）

| ID | 层 | 区域.模块 | 组件/锚点 | 判定 | 设计来源 | 实现位置 | 根因标签 |
|----|----|----------|----------|------|---------|---------|---------|
| SB-L3-01 | L3 | Sidebar.SessionItem | 单行不换行 | ⚠ | sidebar/spec.md §会话项 | components/.../SessionItem.vue:42 | 孤立 |
| ... | | | | | | | |

**判定值**：`✅一致` / `⚠偏差` / `❌缺失` / `🆕多余`（🆕 仅阶段 B 用）

**根因标签**：`孤立` / `疑似根因→<候选>`（如 `疑似根因→StatusDot未抽原子`）

## 二、条目详情卡（仅展开 ⚠/❌/🆕，✅ 不展开）

### [ID] 标题

- **层级位置**：L? · 区域.模块.组件
- **设计要求**：一句话 + 设计来源（`spec路径 §章节` 或 draft html 片段）
- **实现现状**：一句话 + 实现来源（`文件路径:行号` 或 `未找到`）
- **判定**：⚠ / ❌ / 🆇
- **差异描述**：设计要什么，实现实际是什么，差在哪（具体、可复现）
- **设计证据**：原文引用 / class 名 / 状态枚举
- **实现证据**：代码片段（≤10 行）或目录结构
- **初步根因**：孤立问题，还是根因？若根因，列出可能连带的其他条目 ID
- **修复性质**：长期方案 / 短期方案 · 治标 / 治本

## 三、Wave 小结

- 审查条目数：N（✅ x / ⚠ y / ❌ z / 🆕 w）
- 疑似根因候选：列出本 wave 发现的根因聚类苗头
- 跨 wave 依赖提示：本 wave 发现的问题是否需要别的 wave 确认（如"composer 工具条结构依赖 token 是否接入"）

## 四、填写规范

1. **ID 命名**：`<区域缩写>-L<层>-<两位序号>`。区域缩写：SB=Sidebar / WS=Workspace / PN=Panel / OL=Overlays / OV=Overview / ST=Settings / SH=Shell
2. **实现位置必填**：找不到写 `未找到（疑似未实现）`，不要留空。留空 = 未审查。
3. **证据必填**：⚠/❌/🆇 必须双证据（设计 + 实现）。无证据 = 判定无效。
4. **根因标签优先**：宁可多标"疑似根因"，后续阶段 C 会验证收敛。
5. **不修复**：审查阶段只记录，不改代码。
