pi Agent Loop 三层嵌套结构

pi 的 agent-loop.ts 包含三层循环，每层有明确的职责边界。 事件从内到外冒泡，经过 agent-session.ts 转发给 RPC 客户端。

Agent Run (一次 prompt 调用)
Turn (一轮 LLM 调用 + 工具执行)
Stream (一次 LLM 流式响应)
Tool Execution
AGENT RUN

_runAgentPrompt() → agent.prompt() → runAgentLoop()

一次用户 prompt 触发一个 Agent Run。包含一个或多个 Turn。结束时发出 agent_end。

emit agent_start
emit turn_start
emit message_start { role: 'user', content: '...' }
emit message_end
TURN 1

while (hasMoreToolCalls) → streamAssistantResponse() → executeToolCalls()

每个 Turn = 一次 LLM 流式调用 + 工具执行。如果 LLM 输出了 toolUse，执行完工具后 hasMoreToolCalls = true， 进入下一轮 Turn（再次调 LLM）。

STREAM

streamAssistantResponse() → 单次 LLM API 调用

emit message_start { role: 'assistant' }
emit message_update { thinking_start }
emit message_update { thinking_delta: "Let me..." }
emit message_update { thinking_end }
emit message_update { toolcall_start: "read" }
emit message_update { toolcall_delta }
emit message_update { toolcall_end }
emit message_update { toolcall_start: "edit" }
emit message_update { toolcall_end }
emit message_update { text_delta: "I've modified..." }
emit message_update { text_delta: "the file..." }
emit message_end { stopReason: 'tool_use' | 'end_turn' }

如果 stopReason === 'tool_use'，执行工具：

TOOL EXEC
emit tool_execution_start { toolCallId, toolName: 'read' }
emit tool_execution_end { toolCallId, output: '...' }
emit tool_execution_start { toolCallId, toolName: 'edit' }
emit tool_execution_end { toolCallId, output: '...' }
emit message_start { role: 'tool_result' }
emit message_end
emit turn_end

hasMoreToolCalls = true → 继续内层 while → Turn 2

TURN 2
STREAM
emit message_start { role: 'assistant' }
emit message_update { thinking_start }
emit message_update { thinking_end }
emit message_update { text_delta: "Done!" }
emit message_end { stopReason: 'end_turn' }
emit turn_end

stopReason === 'end_turn' → 无 tool calls → 退出内层 while

检查 followUpQueue → 空 → 退出外层 while

emit agent_end
关键认知： message_start/message_end 是 AgentMessage 级别的（LLM 的一个"回复"）， 不是 xyz-agent 前端的一个 Message。pi 的一个 Agent Run 可能产生多个 assistant AgentMessage（多个 Turn）， 但 xyz-agent 前端会把整个 Agent Run 映射为一个 streaming Message。

contentBlocks 如何构建：useChat.ts 中的注册逻辑

xyz-agent 将 pi 的 整个 Agent Run（可能包含多个 Turn、多次 streamAssistantResponse） 合并为前端的一个 streaming Message。contentBlocks 记录这个 Message 内部各 block 的到达顺序。

事件处理函数与 contentBlock 注册

useChat handler 触发条件 contentBlock 操作
onThinkingStart thinking_start 事件 push { type: 'thinking', refId }
仅当上一个 thinking 已 collapsed 时才创建新 block
onToolCallStart tool_execution_start 事件 push { type: 'toolCall', refId: tcId }
onTextDelta text_delta 事件 push { type: 'text', refId: 'text' }
⚠️ 仅首次！之后所有 text_delta 只追加 message.content，不再注册新 block

重要： pi 的 toolcall_start（LLM 输出 tool_use block）和 tool_execution_start（实际执行工具）是两个不同事件。 xyz-agent 只关心执行阶段，所以 EventAdapter 只转发 tool_execution_start/end。 toolcall_start/end 被 EventAdapter 忽略。

以 TCC O TC TC O 为例，分析当前 groupByContentBlocks 的分组逻辑和期望改进方向。

Compact 模式渲染（CompactSummaryBar）

CompactSummaryBar 不用 sections，它直接从 message.thinking/toolCalls 构造 chips：

思考 (1)
read (1)
edit (1)
write (1)
bash (1)

CompactSummaryBar 完全不关心 contentBlocks 中的顺序和位置。它从 message.thinking[] 和 message.toolCalls[] 扁平构造 chip 列表（按 toolName 分组）， 丢失了"哪些 thinking/tool 在 text 前面、哪些在后面"的位置信息。

期望行为

以 text 为分割线，text 之间的所有 thinking + toolCall 合并为一个 "过程" 块。

方案 A：不改数据模型（立即可行）

由于当前只有一个 text block，分组结果为：

方案A
P [thk-1, tc-1, tc-2]
O (全文)
P [thk-2, tc-3, thk-3, tc-4]

text 之前的所有 T/C 合为一组，text 之后的所有 T/C 合为一组。 改动范围：只需改 groupByContentBlocks 的分组算法 + CompactSummaryBar 支持多组渲染。

方案 B：改数据模型（完整方案）

支持多段 text，每段 text 独立记录位置：

方案B
P [thk-1, tc-1, tc-2]
O "I've read..."
P [thk-2, tc-3, thk-3, tc-4]
O "All done!"

需要改动的模块：

模块改动
shared/message.ts Message.content 从 string 改为支持分段存储（如 textSegments: { refId, content }[]）
useChat.ts → onTextDelta text 被打断后恢复时，创建新的 text contentBlock + 新的 text segment
message-layout.ts 以 text block 为分隔线分组（thinking + toolCall 合为 mixed group）
AssistantContent.vue 多段 text + 多个 CompactSummaryBar 交替渲染
CompactSummaryBar.vue 接受指定范围内的 thinking/toolCalls，而非全量
历史消息加载 从 session 文件恢复时重建多段 text

建议路径：先做方案 A（快速验证分组逻辑的正确性和视觉效果）， 确认交互满意后，再做方案 B（支持多段 text）。方案 A 的分组算法改动可以被方案 B 复用。
