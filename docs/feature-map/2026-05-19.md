## 一、愿景与定位

### 一句话定位

基于 pi 的 AI Agent 桌面工作台，核心差异化在于**递归树形 SubAgent 编排 + 多面板 GUI + 用户可介入子任务执行**。

### 核心解决什么问题

1. 当 AI Agent 自动拆分出多个 SubAgent 并行工作时，用户如何直观地看到整体进度？
2. 当某个 SubAgent 需要用户确认时，如何在不影响当前工作流的情况下及时通知？
3. 当用户同时管理多个复杂任务时，如何快速切换上下文而不丢失状态？

### 生态定位

pi 生态中，没有任何项目同时做到：

| 维度 | 现有最佳方案 | xyz-agent 目标 |
|------|-------------|---------------|
| 递归树形 SubAgent（子可生孙） | pi-messenger Crew 是扁平的 planner→workers | 递归深度 20，feedback 通信 |
| 用户可介入执行 + 主 Agent 不感知中间态 | 无（现有方案中断=结束） | RPC 桥接，暂停/重定向/恢复 |
| 桌面多面板 GUI + 任务树可视化 | pi-desktop 无分屏，无树形任务面板 | 分屏、分组、树面板、节点操作 |

## 四、执行顺序

```
Phase 0 (生产可用化)       ← 最先，~3-4 天
  │
  ▼
Phase 1 (基础修复)         ← Phase 0 后，~3-4 天
  │
  ├──→ Phase 2 (聊天增强)  ← 可和 Phase 3 部分并行，~5-7 天
  │       │                  (2.1 @-mention / 2.3 文件变更总览 / 2.5 Session Tree
  │       │                   是 chat 层独立工作，和 drawer 层互不依赖)
  │       │
  ▼       ▼
Phase 3 (SubAgent 数据流)  ← 核心缺口，~5-7 天
  │
  ▼
Phase 4 (高级模式)         ← Phase 3 后，~5-7 天
  │
  ▼
Phase 5 (树形引擎)         ← Phase 4 后，~10-15 天
  │                           需先做 PoC 验证 RPC 桥接可行性（2-3 天）
  ▼
Phase 6 (树形 GUI)         ← Phase 5 后，~7-10 天
```

**总预估**:
- Phase 0-2: ~11-18 天（能实际日常使用）
- Phase 3-4: ~10-14 天（SubAgent 可视化完整）
- Phase 5-6: ~17-25 天（树形引擎差异化）

### 为什么这个顺序

1. **Phase 0 最先**: 无法脱离开发环境运行的产品不是产品。所有后续工作都建立在"能构建出可分发的应用"之上。

2. **Phase 1 先行**: 技术债不修，后续所有功能建立在不可靠的基础上。手动触发路径的 bug 会让 Phase 3 的 TaskTree 数据源不完整。

3. **Phase 2 和 Phase 3 可部分并行**: @-mention、文件变更总览、Session Tree 是 chat 层工作；TaskTree 数据接入是 drawer 层工作。互不依赖。

4. **Phase 3 是 SubAgent 体验的核心缺口**: PRODUCT.md 的核心卖点是"SubAgent 并行状态可视化"，但 TaskTree 和 Drawer 全是静态 mock。

5. **Phase 5 是最大风险项**: RPC 桥接模式需要深入理解 pi `--mode rpc`，递归分解的停止条件需要精心设计。建议 Phase 5 开始前先做 PoC。

6. **Phase 6 依赖 Phase 5 的引擎能力**: SubAgent Tab 栏和子 Agent 对话面板需要 RPC 桥接的事件流。

## 五、关键决策点

### 5.1 已决定

| # | 决策 | 结论 | 理由 |
|---|------|------|------|
| D1 | 技术栈：Electron vs Tauri | **Electron** | 已投入大量开发，Vue 3 + Electron 生态成熟，迁移成本不划算 |
| D2 | 前端框架：Vue vs React | **Vue 3** | 已实现，Pinia + Composition API 体系完整 |
| D3 | SubAgent 执行方式 | **复用 pi subagent extension** | pi extension 已成熟（single/parallel/chain/background），自研无意义 |
| D4 | SubAgent 隔离方式 | **独立 pi 进程** | 进程隔离比同进程多 Session 安全，RPC 协议原生支持交互式控制 |

### 5.2 待决定

| # | 决策 | 背景 | 选项 | 影响 | 建议决策时机 |

### 6.5 pi 事件结构

`tool_execution_start/update/end` 的 payload 结构（来自 `types.ts`）：

```typescript
// tool 开始执行
interface ToolExecutionStartEvent {
  type: "tool_execution_start"
  toolCallId: string
  toolName: string       // "subagent" | "bash" | "read" | ...
  args: any              // subagent: { agent, task, tasks, chain }
}

// tool 执行中流式更新
interface ToolExecutionUpdateEvent {
  type: "tool_execution_update"
  toolCallId: string
  toolName: string
  args: any
  partialResult: any     // subagent 中间输出
}

// tool 执行完成
interface ToolExecutionEndEvent {
  type: "tool_execution_end"
  toolCallId: string
  toolName: string
  result: any            // subagent 最终结果
  isError: boolean
}
```

### 6.6 pi Extension UI 协议

子进程（subagent）通过 `extension_ui_request` 向用户请求输入，外部通过 `extension_ui_response` 回复。这是 Phase 5 RPC 桥接的核心协议：

```typescript
// 子进程 → 外部：请求用户输入
type RpcExtensionUIRequest =
  | { type: "extension_ui_request"; id: string; method: "select"; title: string; options: string[] }
  | { type: "extension_ui_request"; id: string; method: "confirm"; title: string; message: string }
  | { type: "extension_ui_request"; id: string; method: "input"; title: string; placeholder?: string }
  | { type: "extension_ui_request"; id: string; method: "editor"; title: string; prefill?: string }
  | { type: "extension_ui_request"; id: string; method: "notify"; message: string; notifyType?: "info"|"warning"|"error" }

// 外部 → 子进程：回复用户输入
type RpcExtensionUIResponse =
  | { type: "extension_ui_response"; id: string; value: string }    // input/select 的回答
  | { type: "extension_ui_response"; id: string; confirmed: boolean } // confirm 的回答
  | { type: "extension_ui_response"; id: string; cancelled: true }    // 用户取消
```
