# 0019: 核心 User Flow 范围

**状态**: Accepted
**日期**: 2026-06-18
**关联**: ADR-0018（视觉方向）、PRODUCT.md（产品定位）、feature-map

## 背景

feature-map 列了 7 阶段 40+ 功能点，但**功能清单 ≠ 用户旅程**。进入设计阶段前需明确 demo 聚焦哪几条核心 user flow，避免 50 个功能罗列式 demo 失焦。从功能矩阵 + Nielsen 启发式评估痛点 + PRODUCT.md 差异化定位，提炼出 5 条候选 flow。

## 决策

**深度编织 Flow 2 + Flow 3，其余 3 条做入口级轻量展示。**

### 5 条核心 flow

| # | Flow | 类型 | demo 处理 |
|---|------|------|----------|
| 1 | 冷启动 → 下达第一任务 | 入口体验 | 轻量（空状态 + Overview 截图） |
| 2 | 单 Agent 对话 → 代码变更审查 | **日常主路径** | **深度编织** |
| 3 | SubAgent 并行 → 监控 → 介入 | **核心差异化** | **深度编织** |
| 4 | 走错路 → 回退 → 重新对话 | 错误恢复 | 轻量（Session Tree + 分支 pill） |
| 5 | 多任务并行 → 通知 → 收尾 | 长任务 | 轻量（Overview + 通知样例） |

### 选择依据

**Flow 3 必做**：PRODUCT.md 把 SubAgent 并行可视化定为核心卖点，demo 不展示这条等于没讲清楚产品差异化。zcode-demo 已有进程面板 mini chip 探路，但缺 TaskTree 实时生长、并行/链式可视化、请求回应 tab——这是最核心的一块。

**Flow 2 必做**：「友好舒适流畅」目标必须在每天用 100 次的主路径上验证。变更集 Accept/Reject + 消息操作菜单是 critique 点名的 P0 短板（Nielsen #3 用户控制 2/4、#9 错误恢复 2/4）。

**Flow 1/4/5 轻量**：5 条全展开等于做整个产品。这 3 条用空状态、Overview、通知样例带过即可，不编织完整旅程。

## demo 叙事主线

> **一个复杂任务下来 → 主 Agent 拆解 → 子 Agent 并行跑 → 用户随时能看、能插话、能回退 → 最后变更干净落地**

这条主线把"友好（Flow 2 的流畅）+ 差异化（Flow 3 的并行）"串起来：Flow 2 是日常场景铺垫，Flow 3 是高潮展示差异化能力，Flow 4 的回退作为"出错也不怕"的安全网收尾。

## 深度编织产出格式

每条深度 flow 产出三份设计依据（后续画高保真的直接输入）：
1. **屏幕序列** — 每一步在哪个屏幕、看什么、做什么
2. **状态机** — 关键组件/页面的状态及其转换
3. **边缘状态** — 空/错/载/确认/并发等非主路径场景

产出文件：
- `docs/flows/flow-2-code-change-review.md`
- `docs/flows/flow-3-subagent-parallel.md`（下一轮）

## 后续影响

- 高保真设计稿优先覆盖 Flow 2 + Flow 3 涉及的屏幕
- 交互规范文档（Step 5）的状态机/空错载态清单，从这两条 flow 的边缘状态直接生长
- Flow 1/4/5 的轻量展示在 demo 终稿时补，不影响核心设计决策
