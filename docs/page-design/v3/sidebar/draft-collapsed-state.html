h-nav · 26×26×3 · 2px 间距 · ⌘B / ⌘[ / ⌘]
  • 位置：P1 header 最左（traffic-light 右侧 10px 间距）—— 折叠后它是用户第一眼看到的 nav 入口
  • 视觉：三按钮等宽等高，读作一组，位置最左符合视觉起点
  • 状态同步：展开后此按钮变成「收起」图标（▥），与 sidebar 内「收起」nav 项互为镜像

② 左缘唤回条 · 隐性发现入口

2 Header 接管 · 触发条件与视觉细节折叠 + 非全屏 · 第一个 panel header
触发条件：sidebar 折叠 && !全屏 —— 两条件都满足时，chrome 落入 P1 header。
// P1 header 内部布局（折叠 + 非全屏）
.main-header { display: flex; align-items: center; gap: 6px; height: 42px; padding: 0 12px; }
├─ .h-tools { display: flex; gap: 10px; flex-shrink: 0; }
│　├─ .traffic-light { 3 dots · 12px · 8px gap } // 跨平台由应用自绘
│　└─ .h-nav { display: flex; gap: 2px; }
│　　　├─ h-btn: 收起侧栏 (▥) · ⌘B
│　　　├─ h-btn: 后退 (←) · ⌘[
│　　　└─ h-btn: 前进 (→) · ⌘]
└─ .breadcrumb { flex: 1; } // 原内容右移，flex 自动占满剩余
  • 总宽度估算：traffic-light (~52px) + 10px + h-nav (~82px) + 6px gap + breadcrumb = ~150px chrome 区。若 breadcrumb 内容超长会自动 truncate（白空间省略号）
  • 高度由谁决定：42px header 由 h-nav 26px 按钮决定（加 8px 上下 padding）。traffic-light 12px 比 nav 按钮小，视觉重心仍在按钮组
  • 为什么 traffic-light 和 h-nav 之间 10px gap：traffic-light 是 OS 控件，h-nav 是应用控件 —— 10px 间隙明确分组，避免被读作「一组按钮」
  • 面包屑右移的实现：不需要手动 padding-left —— flex:1 自动让 breadcrumb 占满剩余空间，左对齐到 h-tools 之后
  • 跨平台：win/linux 的「最小化 / 最大化 / 关闭」三按钮也由应用自绘（mimic mac），同样 12px + 8px gap；win 上 icon 风格不同（□ ✕ 等）但布局一致
  • win/linux 顺序：右侧三按钮的顺序在 win/linux 是 _ □ ✕，与 mac 的 ● ● ●（左侧）镜像；本稿按 mac 顺序画，win 版镜像即可
3 面板分割 · 只有 P1 header 受影响P2 保持原状 · 不复制 chrome
  • 场景：sidebar 折叠 + 2 个 panel 共存（panel split，详见 workspace/spec.md）
  • P1 header：收到 chrome —— traffic-light + 折叠 + ←/→ + breadcrumb1（P1 自己的面包屑反映自己的会话上下文）
  • P2 header：保持原状 —— 只有 breadcrumb2，不重复 chrome
  • 为什么不是两个 header 都有 chrome：
    • traffic-light 是 window-level（关/最小化/最大化的是整个 window）—— 它属于一个地方，放在 P1 是「视觉起点 + 用户第一眼」的位置
    • 折叠 / ←/→ 是 sidebar / window-level，不是 per-panel —— 复制到 P2 会冗余且误导（用户不知道按哪个）
    • 每个 panel 自己的 breadcrumb是 per-panel 信息（反映 P1 / P2 各自的会话上下文），所以两个 panel 都有且应该都有
  • 视觉不对称：P1 header 比 P2 header 宽（左侧多了 chrome 区域）—— 这是预期且必要的不对称，反映 chrome 的归属层级（window > panel）
  • 3 个 panel 怎么办：仍只 P1 有 chrome，P2 / P3 都没有 —— chrome 是 window 的属性，不是 panel 列表的属性
4 全屏态差异 · 无 traffic-lightOS 自动隐藏 · h-nav 仍在
  • 触发：sidebar 折叠 + 全屏（mac：⌃⌘F / win：F11）
  • traffic-light 行为：mac 自动隐藏（OS 行为，不是应用决定）；win/linux 上若应用自绘，也应隐藏（避免和 OS 任务栏冲突）
  • header 差异：P1 header 的 .h-tools 区只包含 .h-nav（折叠 + ←/→），无 .traffic-light
  • 视觉：P1 header 左侧「少了」50px chrome 区，breadcrumb 整体左移 50px —— 视觉上是「更轻」的 header
  • 布局稳定性：header 高度仍 42px，flex 布局不变 —— breadcrumb 自然延伸，无 jank
  • 不需要占位符：traffic-light 位置不留空白分隔（不是 toggle 显隐的 chip，是 OS 隐藏）—— 直接让 h-nav 紧贴左边

6 状态保留 · 折叠 / 唤回不丢上下文tab · active session · scroll · 视图节点 · chrome 状态
  • tab 状态：session list ↔ file view 切换记入 Sidebar 容器状态；折叠不影响，唤回后精确恢复
  • active session：折叠时 active 项冻结；唤回后 session list 仍定位到该项（包括 scroll 位置）
  • main-panel 内容：折叠 ≠ 清屏，main-panel 一直显示当前 session 的工作区
  • ←/→ 导航栈：与 sidebar 状态正交 —— 折叠时栈不丢，唤回后仍可 ←/→ 回退
  • chrome 状态：折叠后 chrome 在 P1 header，始终可见（无 toggle 显隐）—— 不需要「记忆」chrome 状态
  • 滚动位置：sidebar 内的 session list / file tree 滚动位置不重置；main-panel 内的消息流 / composer 滚动位置不重置
待验证 / 后续细化：
  • 左缘唤回条：chrome 已在 P1 header 后，左缘条还是否必要？当前保留为「次级兜底」—— 是否降级为「右键菜单里藏」以减少视觉噪声？
  • P1/P2 header 视觉不对称：P1 header 比 P2 header 宽（左侧 chrome 区域）—— 是否需要用一条 1px 分隔线在 panel 之间明确「这是两个独立 panel」？当前 panel border 已经隐含了
  • 全屏下 traffic-light 位置：mac OS 自动隐藏，但 win/linux 上若自绘需要明确「全屏时不绘制」—— 应用代码要带 if (!fullscreen) renderTrafficLight() 守卫
  • 3+ panel 的 P1 chrome 归属：当前设计是「P1 有，P2/P3 没有」。但 P1 的折叠按钮折叠的是 sidebar，不是 panel —— 语义上 P1 的 chrome 是 window-level，不是 P1 专属。这个「为什么是 P1 不是 P2」的解释需要写进 spec
  • win/linux tooltip：当前是「展开侧栏」+「Ctrl+B」；是否需要单独加平台标注（mac icon ⌘ / win icon Ctrl）？
