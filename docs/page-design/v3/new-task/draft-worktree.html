新建任务 · worktree 创建

bare repo + worktree 模式下，在选目录 popover 里加一条「新建 worktree」动作项。 点击后弹 modal 填参数（分支名 + base），创建完立即生效——directory/branch chip 自动切到新 worktree 的值， popover 关闭，用户回到 landing 态继续写消息。worktree 创建是独立的前置步骤，不与消息发送耦合。

位置 DirSelectPopover 动作项 执行 eager（创建完立即生效） 适用 bare repo + worktree（.bare 模式）

入口设计裁决： ① 放在 DirSelectPopover 动作项区——worktree 本质是「选一个目录」，与「打开文件夹」（选本地已有目录）、「远程连接」（选远程目录）是平行路径。心智模型一致：确定在哪个目录工作； ② accent-soft 底色——区别于其他两个常规动作项（透明底）。表达「这是 .bare 模式的扩展能力」，视觉上可快速定位，但不喧宾夺主（列表项仍是主角）； ③ 条件显示——仅当 gitIsWorktree === true 或向上能找到 .bare/ 时显示。非 bare repo 项目不显示此动作项，其他两个动作项正常； ④ 文案「新建 worktree…」——省略号表示「会弹框填参数」（与「打开文件夹…」同语言），动词开头（design-system §9）。

§2CreateWorktreeModal · 三态

点击「新建 worktree…」→ 居中 modal（不可逆动作必须 modal，参 spec §2.3）。三态：form（填参数）→ progress（创建中 10-30s）→ success（成功，2s 自动关闭）。创建完立即生效，directory/branch chip 切到新值。

三态裁决： ① form：分支名 + base branch + 实时预览目录名。base 默认 origin/main（与 git-cwt 一致）； ② progress：顶部 indeterminate loading bar + head icon 变 spinner + 3 步固定列表（创建分支 → 检出目录 → 运行 setup 脚本）。前两步通用，第三步是项目脚本黑盒（运行 .bare/custom-hooks/setup-worktree.sh），用户可折叠查看实时 stdout/stderr。不可配置，降低认知负荷。创建中 modal 不可关闭（无 X，foot 只有 disabled「创建中…」按钮），强制等待完成或失败； ③ success：2s 后自动关闭。关闭的同时 flow.selectWorkspace(newCwd) 被调用，directory chip 切到新 worktree，branch chip 经 git info 刷新切到新分支。用户回到 landing 继续写消息； ④ 失败：不自动清理已创建的 worktree（遵循 AGENTS.md 规则 #0）。默认按钮是「重试 setup 脚本」不是「清理」（破坏性操作默认偏保守）。失败信息含脚本退出码 + stderr 输出，便于定位。
setup 脚本契约（方案 A）： ① 脚本路径固定：<wsRoot>/.bare/custom-hooks/setup-worktree.sh（git-cwt 约定）； ② 脚本内容是项目黑盒——xyz-agent 不假设项目用什么包管理器、构建工具、缓存机制。脚本可能是 pnpm install（前端）、uv sync（Python）、cargo build（Rust），或完全自定义（如 xyz-agent 自己的脚本还会复用 Electron/pi 二进制缓存）； ③ 脚本不存在时跳过——步骤 3 显示「未找到 setup 脚本，已跳过」+ worktree 直接可用（但没装依赖，用户需手动）。xyz-agent 不做 fallback 项目类型检测（pnpm/npm/uv/cargo 全家桶识别不全且永远是坑）； ④ 实时输出：脚本 stdout/stderr 实时 tee 到 log-box，用户可折叠查看。完成态保留输出供失败时排查。

§3创建后立即生效

modal 关闭（success 自动 / 用户点「开始」）后，同一画面的 directory/branch chip 已更新为新 worktree 的值。 用户无需任何额外操作，直接在 composer 写消息发送，走正常 submitFirstMessage（pendingCwd 已是新 worktree 路径）。

§4选目录的三条路径

DirSelectPopover 的三个动作项，本质是「确定在哪个目录工作」的三种方式：

路径动作适用场景执行方式
列表选已有点列表项最近用过的 workspace / worktree即时（仅写 pendingCwd）
打开文件夹OS 原生 dialog选本地任意目录（不在最近列表里）即时（OS dialog → pendingCwd）
新建 worktreemodal 填参数 → 创建.bare 模式下派生新工作目录异步（git worktree add + setup 10-30s → pendingCwd）
三条路径终点一致：都是写 pendingCwd，landing 态后续行为完全相同。 区别只在「目录怎么来的」——已存在（列表/dialog）vs 新建（worktree）。 这正是把 worktree 放在 DirSelectPopover 的根本理由：它就是「选目录」的一种方式。

§6边界与错误矩阵

场景处理
非 bare repo「新建 worktree」动作项不显示（其他动作项正常）
modal 里分支名留空「创建」按钮 disabled
分支已存在但无 worktreegit worktree add <path> <branch>（检出已有分支，不 -b）
目录已存在（同名 worktree）modal 报错 + 提示「该 worktree 已存在」→ 关闭后走列表选已有
setup 脚本不存在步骤 3 显示「未找到 setup 脚本，已跳过」+ worktree 直接可用（无依赖，需手动装）。不做 fallback 项目类型检测
setup 脚本失败（非 0 退出）modal 转 error 态 + 显示退出码 + stderr 输出。保留 worktree 目录。默认按钮「重试 setup 脚本」
setup 超时（120s）杀进程 + 报错 + 保留目录
创建中用户关 modalprogress 态 X 按钮隐藏，不可关闭。必须等完成或失败

§7不做的事（YAGNI）

  • worktree 管理 UI（列表/删除/切换）—— separate feature
  • fallback 项目类型检测（检测 package.json / pyproject.toml 自动装依赖）——识别不全且永远是坑。setup 脚本不存在时直接跳过，让用户自己装
  • streaming 实时日志——MVP 用固定步骤列表。完整版后续改 WS streaming
  • 非 bare repo 的通用 git worktree add——场景不同，不在本方案
  • worktree 间 session 迁移——session cwd 固定，迁移需 fork
