§1结构 · 工具区右对齐、无边框、文字融合

composer-box 是单一容器（同一 --bg-input 底 / --border / --radius-lg）。输入区上、工具条下，中间无强分隔线。工具条新规：左侧只留 + 添加；右侧依次是「上下文 · 模型 · 思考」，三者无边框、纯文字、底色透明，hover 才浮一层 --bg-hover——与 composer 底融为一体；发送按钮是唯一实色，最右锚定。

⏎ 发送 · ⇧⏎ 换行 · @ 引用 · # 文件 · / 命令
①+ 添加 —— 左锚定。点开浮层含「附件 / @ 引用 / # 文件 / / 命令」四路；也可直接在输入区敲对应符号触发。/ 需在输入流最左且左侧无内容时触发，右侧内容作过滤 query，⏎/Tab 选中确认。
②上下文（token 容量） —— 文字显当前用量 6.9万 · 6.9%，hover 出容量 popover（已用/总量/百分比/缓存命中率）。用量高自动转 warning 色。详见 §2a。
③模型 —— click 出 provider 分组 select（Anthropic / OpenAI / Google …，非级联，平铺单层选）。详见 §2b。
④思考等级 —— click 出 6 级（off/low/medium/high/xhigh/max），可用等级随模型变。级别越高越偏紫 + 发光，max 变实色块。详见 §2c。
⑤发送 —— 最右、唯一实色。空态禁用并退成透明文字；AI 工作中换「停止」；发送中换 spinner。
「与背景融合」的 4 条规则：① 上下文/模型/思考共享 .c-text 类——border:0、background:transparent、文字 --text-tertiary；② hover 才浮一层 --bg-hover，移开即隐；③ 三者间隙靠 gap:2px + 各自 padding，不画分隔线；④ 发送是唯一的实色锚点，把右侧三颗文字按钮「压」成一组。
舒适流畅：所有态切换 220ms cubic-bezier；聚焦 ring 作用于整个 composer-box（不抖盒模型）；按钮无硬边框，视觉上输入区与工具区是同一张皮。
输入区与工具栏 4px 间隔（composer-bar mt-1）：偏离原「同一张皮」无 gap，加 4px 呼吸，保留无分隔线的融合观感（用户拍板）。

§2工具区浮层 · 6 个

逐一设计 composer 工具条会弹出的 6 个浮层：上下文容量、模型分组、思考等级、@ 引用、# 文件、/ 命令。前三个由工具按钮触发；后三个由输入区符号触发。所有浮层都向上开、归属 composer、生命周期绑 composer（失焦 / esc 即销毁），不归全局 Overlay。

§2e slash 命令 · 内联 token（非附件 chip）

/ 命令选中后直接插在输入流内（不是浮在上方的附件 chip），是一个紫色整体 chip——可 backspace 整体删除，删除时光标停在 chip 后。输入区是 contenteditable 以支持内联富文本。

/fix× 把 AuthService.login 改成异步，返回 Promise<Token>
slash-token 行为：① 整体一个 chip，user-select:none——backspace 一次删整块；② chip 内带 × 可点删；③ chip 必须在输入流最前（slash 命令是「操作模式」前缀），其后接普通文本作为该命令的参数；④ 同一输入只允许一个 slash-token，再敲 /（已有内容时）不触发命令浮层，作普通字符。

§4steer / followup · 两色区分

两者底层调用不同 runtime 方法（steer 追加进当前回合；followup 在当前回合结束后开新轮），语义不同——用户应能感知「我这条是追加，还是开新的」。同族虚线+脉冲气泡保持「待发送」整体感，靠颜色 + WHO 标 + 副文案三维度区分，不靠形状/位置分叉（避免割裂输入区节奏）。提交共三种操作（steer / followup / 立刻打断，见下触发表）；已排队消息按底层双 FIFO 分组展示（steering 先于 followUp，见 §5）。

提交操作 · steer/followup（仅键盘）

S6（AI 执行中）下用户有两种提交意图，靠修饰键区分，温和追加、不打断当前回合：steer（⏎·追加进当前回合）/ followup（Alt+⏎·回合后开新轮）。底层是两个独立 FIFO 队列，steering 整体优先于 followUp（详见 research/pi-steer-followup-capability.md）。立刻打断（abort 喊停）暂不实现——abort 不清队列会复活 steer，见下方警示。

队列模式 · all（高级 toggle，默认隐藏）

底层 drain 有两种模式：one-at-a-time（默认·每条独立一轮 LLM·每条得独立响应·token 开销大）与 all（多条合并成一次 LLM 调用的连续 user message·省 token 但无逐条响应）。普通用户无需理解此区别，默认隐藏；作为高级 toggle 暴露（场景：用户明确想「一次性把几个补充都给 LLM 看」）。toggle 位在 composer 高级设置（不在主工具条），开启后队列气泡标注「合并模式」。

§5双队列视图 · steering / followUp 分组（只读）

底层是两个独立 FIFO 队列：steeringQueue（steer）与 followUpQueue（followup），drain 时 steering 整体优先于 followUp——只要 steering 还有货，followUp 永不被消费。用户交错发 steer(A)→followup(B)→steer(C)→followup(D)，实际生效顺序是 A→C→B→D（非输入顺序）。所以队列视图必须双队列分栏，不能用单 FIFO——否则会掩盖 steer 的插队语义，用户误判生效顺序。

双队列分组：steering 组在上（先生效），followUp 组在下（后生效），虚线分隔。每组内严格 FIFO。为何分栏：steer 整体插队是底层硬语义，单 FIFO 视图会掩盖「steer 永远排在所有 followUp 前」，导致用户误判生效顺序。
实时同步：订阅 queue_update 事件维护本地镜像 steering[] / followUp[]；已 drain 的（被 message_start 消费）从镜像 splice，实时反映「还剩几条」。
⚠️ 当前决定：只读展示（2026-06）：待生效队列在 RPC 下只能入队 + 可见，无法清除/撤回/改类型——session.clearQueue() 是 TUI 专属（仅 handleDequeue 调用），RPC 无 clear_queue 命令（grep RpcCommand 全空），亦无单条删除 API（PendingMessageQueue 无 removeAt）。若提供 dequeue 按钮，点了服务端队列清不掉，下次发送旧 steer 会静默复活 + 新输入双重注入——是假按钮。所以 composer 不提供任何队列写操作入口（无 dequeue / 编辑 / 删除），只靠 queue_update 事件镜像只读展示。详见 research/pi-steer-followup-capability.md。

队列只读 · 入队后不可改

pi 底层队列写能力是全有或全无且 TUI 专属——只有 clearQueue()（全清），没有单条删除 API。RPC 连全清都做不到。所以 composer 当前版本：只入队、只读展示，入队后不可修改/撤回/改类型。已 drain 进 LLM 的消息完全不可撤回。

可见：订阅 queue_update 事件，steering/followUp 数组全量覆盖本地镜像，入队/出队/清空都实时同步。
不可写：无 dequeue / 编辑 / 删除按钮，待发送气泡（单条 pending / 多条 queue）均只读，入队即不可改。
已 drain 的不可撤回：消息一旦进 LLM 上下文（message_start 后）就是历史。
长期 TODO（后端先行）：扩展 RPC clear_queue + get_queued_messages，PendingMessageQueue 加 removeAt(i)、AgentSession 加 removeLastSteering() 等，才能做可靠的撤回/草稿管理。

§6输入暂存 · 切换 session/panel 不丢失

composer 的全部输入态按当前 session（及 panel）维度暂存，切换走开再回来，原样恢复。这是「舒适流畅」的基础——用户不该因为切个 tab 就丢草稿。

暂存项粒度恢复时机
输入文本（含 slash-token / 内联 @ #）per session切回该 session 即恢复光标位
已附上下文（@ / # / 附件）per session同上，附件引用一并恢复
模型选择per session每 session 记各自模型
思考等级per session同上
slash-token（/fix 等）per session随文本恢复
待发送队列（steer/followup）per session队列本身在 runtime，composer 仅镜像显示
实现：composer 维护一个 draftStore[sessionId]，输入/附件/模型/思考每次变更即写（debounce 200ms）；panel 切换时卸载当前、载入目标 session 的草稿。不跨 session 共享——每个 session 是独立草稿。清空时机：消息成功入流后清该次草稿（队列态除外）；显式「清空」按钮可手动清。
