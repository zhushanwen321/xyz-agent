缺失ask-user — 自定义交互组件 ctx.ui.custom() → composer overlay RPC no-op

ask-user extension 用 ctx.ui.custom()（src/index.ts:132）传入自定义 TUI 组件 AskUserComponent，inline 渲染在对话流中——多问题 tab、选项导航、freeform 编辑、Submit 确认。 ctx.ui.custom() 在 RPC 模式下返回 undefined（no-op），整个交互无法触达前端。 ask-user 的 execute 拿到 undefined 后 result.questions 抛 TypeError → 被 try/catch 吞掉 → 返回 isError。用户看不到任何 UI，LLM 收到错误以为工具不可用。

实现路线：xyz-agent 前端在 composer 上方渲染 ask-user 富组件（overlay），用户回答后通过 dialog 协议（extension_ui_response）回传给 pi 的 execute。 前提是 ask-user 改用 ctx.ui.select() 序列而非 ctx.ui.custom()，或 xyz-agent 前端拦截 ask_user tool 调用后自行编排。

当前RPC 下完全不可用 ctx.ui.custom → undefined

无 UI + TypeError
ctx.ui.custom() 返回 undefined
→ result.questions 抛 TypeError
→ try/catch 吞掉 → isError: true
→ LLM 收到错误，以为工具坏了
根因：ctx.ui.custom() 在 RPC 模式下是 no-op（rpc-mode.js:151-154 返回 undefined）。 ask-user 的全部富交互（AskUserComponent 的 render/handleInput）没有任何 RPC 通道能触达前端。 ctx.hasUI 在 RPC 下为 true，所以 headless 检查不触发，代码走到 ctx.ui.custom() 就废了。

实现路线对比

路线 机制 优点 代价
A. 前端拦截
推荐		 前端识别 ask_user tool 调用，自行渲染 composer overlay，答案通过 tool result 注入 不改 pi 端；完整保留富交互（tab/freeform/comment） 需绕过 pi 的 execute 阻塞模型；答案注入机制需设计
B. 降级 dialog ask-user 改用 ctx.ui.select() 序列替代 ctx.ui.custom() 走标准 dialog 协议，前端只需实现 ExtensionUIDialog 需改 extension 源码；丢失 tabbed 多问题、分屏、inline 编辑器体验
C. 协议扩展 Pi RPC 加自定义 method 透传 ctx.ui.custom 的渲染数据 通用，所有 ctx.ui.custom 的 extension 都受益 需改 pi runtime；Component 的 render 输出是 ANSI string[]，前端仍需解析
推荐路线 A（前端拦截）：不依赖 pi 端改动，在 xyz-agent 前端拦截 ask_user tool 调用。 ask-user 的 input schema（Question[]：header/options/multiSelect/allowComment/allowOther）是结构化 JSON，前端可直接消费渲染富组件。 核心问题是答案回传——pi 的 execute 在 await ctx.ui.custom() 处阻塞，前端需通过某种通道把答案注入。 最干净的做法是让 xyz-agent 的 runtime 层在检测到 ask_user tool 时，替它构造 tool result 返回给 pi（截断 execute 的阻塞）。
